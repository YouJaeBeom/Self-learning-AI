# 2. 보안 집계, 그리고 연합 + 파운데이션 모델(LLM)

기초편에서 "원본을 안 보내도 업데이트로 정보가 샐 수 있다"고 했습니다. 이 글은 그 누수를 막는 보안 집계를 자세히 보고, 2026년의 가장 뜨거운 흐름 — 거대 LLM을 연합으로 파인튜닝하기 — 까지 잇습니다. 연합학습이 "원본을 안 보낸다"는 것만으로는 프라이버시를 완전히 보장하지 못한다는 사실에서 출발해, 그 빈틈을 메우는 암호 기법과 LLM 시대의 확장을 다룹니다.

## 프라이버시를 진짜로, 그리고 LLM 시대로

두 주제를 봅니다. 보안 집계(Secure Aggregation)는 서버조차 개별 클라이언트의 업데이트를 못 보게 만드는 암호 프로토콜입니다. 연합 + 파운데이션 모델은 수십억 파라미터 LLM을, 데이터를 모으지 않고 여러 기관이 함께 파인튜닝하는 것입니다. 둘은 연합학습을 실용 단계로 끌어올리는 두 축입니다.

## 보안 집계

문제는 이렇습니다. 서버가 클라이언트별 업데이트를 그대로 받으면, 그것을 분석해 원본 데이터를 추론할 수 있습니다(gradient inversion). 모델 업데이트에는 그 데이터의 흔적이 남기 때문입니다. 서버를 완전히 믿을 수 없을 때, 어떻게 합계만 알게 할까요?

핵심 아이디어는 클라이언트들이 서로 상쇄되는 무작위 마스크를 업데이트에 더해 보내는 것입니다[1]. 개별값은 마스크로 가려져 알아볼 수 없지만, 모두 합치면 마스크가 상쇄되어 진짜 합계만 남습니다.

```
클라이언트 A: 업데이트_A + 마스크_A
클라이언트 B: 업데이트_B + 마스크_B   →  합산  →  업데이트_A+B (+ 마스크들이 상쇄)
클라이언트 C: 업데이트_C + 마스크_C
                         서버는 합계만 본다
```

서버는 합쳐진 결과만 봅니다. 누가 무엇을 보냈는지는 모릅니다. 그리고 클라이언트가 중간에 이탈(dropout)해도 복구되도록 설계됩니다. 현실의 휴대폰은 자주 빠지기 때문입니다. 이것이 Bonawitz 등(2017)의 Secure Aggregation이며, 차등 프라이버시(잡음 추가)와 함께 쓰면 프라이버시가 더 강해집니다[3].

## 왜 연합 + LLM인가

LLM은 거대한 공개 데이터로 사전학습됩니다. 하지만 진짜 가치 있는 데이터 — 병원 기록, 기업 내부 문서, 법률 자료 — 는 모을 수 없습니다. 프라이버시·규제·영업비밀 때문입니다. 연합학습은 바로 이 데이터로 LLM을 파인튜닝할 길을 엽니다. 데이터를 한곳에 모으지 않고도 여러 기관의 지식을 모델에 담을 수 있는 것입니다.

도전은 규모입니다. LLM은 너무 커서 기초편의 FedAvg를 그대로 쓰면 통신·계산이 폭발합니다. 수십억 파라미터를 매 라운드 주고받는 것은 불가능합니다. 핵심 해법이 PEFT(파라미터 효율적 파인튜닝)입니다. 거대 모델 전체가 아니라 작은 어댑터(LoRA 등) 파라미터만 학습하고 주고받습니다[2]. 통신량이 수백에서 수천 분의 1로 줄어 연합이 현실적이 됩니다. S7에서 본 LoRA가 여기서 연합학습의 통신 문제를 푸는 열쇠로 다시 등장합니다.

## FederatedScope-LLM

FederatedScope-LLM(Kuang et al., 2024)은 "LLM을 연합으로 파인튜닝"을 위한 종합 패키지입니다[4]. PEFT 통합으로 LoRA 등 효율적 파인튜닝을 연합 환경에 결합해 통신을 줄입니다. 벤치마크·파이프라인으로 연합 LLM 파인튜닝을 재현하고 비교할 수 있는 데이터셋·평가 틀을 제공합니다. 실용 도구로 이질적 클라이언트, 리소스 제약 등 현실 문제를 다루는 엔지니어링을 담습니다. 의의는 연합학습이 "작은 모델"을 넘어 파운데이션 모델 시대로 들어섰음을 보여주는 대표 실용 프레임워크라는 것입니다.

## 봉투 합산과 매뉴얼의 '수정 스티커'

두 개념을 일상에 빗대면 직관이 잡힙니다. 보안 집계는 "용돈 봉투 합산"과 같습니다. 각자 자기 값을 가린 채 내고, 합쳐야 의미가 나옵니다. 누구도 남의 봉투 속을 보지 못하지만, 다 모으면 총액은 정확합니다. 연합 LLM(PEFT)은 두꺼운 표준 매뉴얼(LLM) 전체를 다시 인쇄해 주고받는 대신, "여기 이렇게 고치세요"라는 얇은 수정 스티커(LoRA)만 교환하는 것입니다. 본문은 그대로 두고 스티커만 모아 붙입니다. 무거운 본체는 고정하고 작은 변화량만 오가게 하는 발상입니다.

## 수식의 의미

보안 집계는 `Σ_k (업데이트_k + 마스크_k) = Σ_k 업데이트_k`로, 마스크들이 쌍으로 상쇄되어 서버는 합계만 봅니다. PEFT 연합은 학습·전송 대상이 전체 W가 아니라 작은 어댑터 `ΔW = AB`(저랭크)뿐이라 통신량이 급감합니다. 한 줄로 말하면 보안 집계는 "가려도 합치면 드러난다"이고, PEFT 연합은 "전체 대신 작은 변화량만 오간다"입니다.

## 한계와 주의

보안 집계의 비용·복잡성이 있습니다. 암호 연산·dropout 처리로 오버헤드가 크고, 클라이언트 수가 많을수록 설계가 까다롭습니다. 차등 프라이버시(DP)의 정확도 손실도 있습니다. 잡음이 강하면 모델이 나빠집니다. 프라이버시와 정확도의 맞교환입니다. 연합 LLM의 난점도 남습니다. PEFT로도 여전히 무겁고, non-IID·이질 리소스·데이터 품질 문제가 그대로 남습니다. 공격은 진화합니다. 보안 집계와 DP도 만능이 아니며, 위협 모델을 명확히 하고 설계해야 합니다.

## 프라이버시는 층으로 쌓는다, 그리고 위협 모델

연합학습의 프라이버시는 단일 장치가 아니라 여러 층으로 쌓입니다. 첫 층은 연합 자체입니다. 원본 데이터를 보내지 않고 모델 업데이트만 보내니, 데이터가 기기를 떠나지 않습니다. 하지만 앞서 봤듯 업데이트에도 데이터의 흔적이 남아, 이것만으로는 부족합니다. 둘째 층이 보안 집계입니다. 상쇄 마스크로 서버조차 개별 업데이트를 못 보게 합니다. 셋째 층이 차등 프라이버시(DP)입니다. 업데이트에 정교하게 설계된 잡음을 더해, 특정 개인의 데이터가 학습에 쓰였는지조차 통계적으로 알 수 없게 만듭니다.

여기서 핵심 트레이드오프가 차등 프라이버시의 잡음입니다. 잡음을 많이 더할수록 프라이버시는 강해지지만 모델 정확도가 떨어집니다. 너무 강하면 모델이 쓸모없어지고, 너무 약하면 프라이버시 보장이 의미 없어집니다. 그래서 "얼마나 강한 프라이버시를 얼마의 정확도와 맞바꿀 것인가"를 정하는 것이 연합학습 설계의 중요한 결정입니다. 이 강도는 보통 수학적으로 정량화되어, "이 정도 잡음이면 한 개인의 기여를 이만큼 가린다"는 보장으로 표현됩니다.

마지막으로, 어떤 프라이버시 기법도 위협 모델을 명확히 해야 의미가 있습니다. 누구를 못 믿는가에 따라 필요한 방어가 다릅니다. 서버를 못 믿으면 보안 집계가 필요하고, 다른 클라이언트를 못 믿으면 또 다른 방어가, 최종 모델에서 정보가 새는 것을 막으려면 차등 프라이버시가 필요합니다. "무조건 안전하다"는 없고, "이런 공격자에 대해 이렇게 안전하다"만 있습니다. 그래서 연합학습 시스템을 설계할 때는 먼저 무엇으로부터 무엇을 지킬 것인지를 분명히 하고, 그에 맞는 방어를 층으로 쌓아야 합니다.

## 정리

보안 집계는 상쇄 마스크로 '서버조차 개별 업데이트를 못 보게' 하고, 연합 + LLM은 PEFT(작은 어댑터만 교환)로 거대 모델을 데이터 없이 함께 파인튜닝합니다(FederatedScope-LLM). 연합학습이 작은 모델을 넘어 파운데이션 모델 시대로 들어선 것입니다. 이로써 07 연합학습 트랙의 두 글이 세 도전에서 보안·연합 LLM까지 이어졌고, 심화편 태스크별 방법론 전체가 마무리됩니다.
