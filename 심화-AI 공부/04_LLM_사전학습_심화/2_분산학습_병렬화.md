# 2. 분산학습과 병렬화 — 한 GPU로는 안 되는 이유

수천억 파라미터 모델은 한 GPU에 들어가지조차 않습니다. 모델 가중치만으로도 GPU 메모리를 넘치고, 학습에 필요한 부수 데이터까지 더하면 한참 모자랍니다. 그래서 거대 모델 학습은 수백에서 수천 개의 GPU에 일을 쪼개 나눠 주는 공학입니다. 이 글은 그 '쪼개는 세 가지 방법'과, 메모리를 아끼는 ZeRO를 직관으로 봅니다. 핵심을 미리 말하면, 마냥 쪼갠다고 빨라지는 것이 아니라 통신 비용이라는 대가가 따른다는 점입니다.

## 안 들어가는 모델을 어떻게든 학습시키기

문제는 두 가지입니다. 하나는 메모리입니다. 모델 가중치에 더해 기울기, 옵티마이저 상태(Adam의 m과 v, S1), 활성값까지 한 GPU에 다 들어가지 않습니다. 특히 Adam은 파라미터마다 두 개의 모멘트를 더 저장하므로, 옵티마이저 상태만으로도 모델 크기의 몇 배 메모리를 먹습니다. 다른 하나는 시간입니다. 수조 토큰을 한 GPU로 처리하면 수십 년이 걸립니다.

목적은 일을 여러 GPU에 쪼개 메모리도 나누고 속도도 끌어올리는 것입니다. 단, GPU끼리 소통하는 통신 비용이 들기 때문에 무작정 쪼갠다고 빨라지지는 않습니다. 이 균형을 잡는 것이 거대 학습 엔지니어링의 핵심입니다.

## 세 가지 병렬과 ZeRO

쪼개는 방법은 무엇이 부족한가에 따라 세 갈래로 나뉩니다.

데이터 병렬(data parallelism)은 같은 모델, 다른 데이터 방식입니다. 모델 사본을 각 GPU에 복제하고 데이터 묶음을 GPU마다 다르게 줍니다. 각자 기울기를 구한 뒤 평균 내어 똑같이 갱신합니다(동기화). 가장 단순하지만, 모델이 한 GPU에 들어가야 한다는 한계가 있습니다. 복제하는 방식이기 때문입니다. 그래서 거대 모델에는 이것만으로 부족합니다.

텐서 병렬(tensor parallelism)은 한 층을 쪼개는 방식입니다[1]. 하나의 행렬 곱(Wx, 기초편 1단계)을 여러 GPU가 조각내어 동시에 계산합니다. 한 층이 너무 커서 한 GPU에 안 들어갈 때 그 내부 연산을 가로세로로 나눕니다. 단점은 매 층마다 GPU 간 잦은 통신이 필요하다는 것이라, 보통 같은 서버 안의 고속 연결로 묶인 GPU끼리 씁니다.

파이프라인 병렬(pipeline parallelism)은 층을 나눠 맡기는 방식입니다[3]. 모델의 층들을 GPU별로 나눠 배치합니다(1~8층은 GPU0, 9~16층은 GPU1 식으로). 데이터가 컨베이어 벨트처럼 GPU를 차례로 통과합니다. 단점은 앞 단계가 끝나길 기다리는 빈 시간(버블)이 생긴다는 것이라, 묶음을 잘게 흘려보내 완화합니다.

마지막으로 ZeRO는 중복을 제거해 메모리를 절약합니다[2]. 데이터 병렬은 모든 GPU가 같은 것(모델·기울기·옵티마이저 상태)을 중복 저장해 낭비가 큽니다. ZeRO(Zero Redundancy Optimizer)는 이 중복을 없애, 각 GPU가 전체의 일부 조각만 저장하고 필요할 때 주고받습니다. 데이터 병렬의 편함을 누리면서 메모리를 크게 절약해 훨씬 큰 모델을 같은 하드웨어로 학습할 수 있습니다. 2026-06 기준 실제 학습은 이 셋과 ZeRO를 섞어(3D 병렬) 씁니다.

| 방식 | 무엇을 쪼개나 | 푸는 문제 | 대가 |
|---|---|---|---|
| 데이터 병렬 | 데이터 | 데이터가 너무 많음 | 모델은 한 GPU에 들어가야 |
| 텐서 병렬 | 층 내부 연산 | 한 층이 너무 큼 | 잦은 통신 |
| 파이프라인 병렬 | 층 | 모델이 너무 깊음 | 버블(대기 시간) |
| ZeRO | 모델·기울기·옵티마이저 상태 | 메모리 중복 낭비 | 조각 주고받는 통신 |

## 거대한 책을 함께 번역하기

100명이 수천 쪽짜리 책을 번역한다고 합시다. 한 명이 다 못 합니다. 데이터 병렬은 똑같은 사전을 가진 번역가 100명이 각자 다른 챕터를 번역하는 것입니다. 사전이 각자 책상에 다 있어야 하므로 모델 복제에 해당합니다. 텐서 병렬은 한 문장이 너무 길어 여러 명이 그 문장의 다른 부분을 동시에 번역하고 합치는 것입니다. 자주 상의해야 하므로 잦은 통신입니다. 파이프라인 병렬은 번역을 공정으로 나누는 것입니다. A는 초벌, B는 윤문, C는 교정을 맡고 원고가 차례로 흘러갑니다. 앞 공정을 기다리는 시간이 버블입니다. ZeRO는 모두가 두꺼운 사전 전체를 복제하는 대신, 사전을 조각내 나눠 갖고 필요할 때 빌려 보는 것입니다. 책상(메모리) 공간을 절약합니다.

## 무엇을 어느 축으로 쪼개나

```
무엇이 부족한가?
 ├─ 데이터가 너무 많다       → 데이터 병렬 (모델 복제, 데이터 분담)
 ├─ 한 층이 너무 크다        → 텐서 병렬 (층 내부 연산 분담)
 ├─ 모델이 너무 깊다(층 많음) → 파이프라인 병렬 (층을 나눠 맡김)
 └─ 메모리 중복이 아깝다     → ZeRO (모델·기울기·옵티마이저 상태를 조각내 분산)
실제 대형 학습: 이 넷을 조합(3D 병렬 + ZeRO)
```

핵심 트레이드오프는 항상 통신 대 메모리·연산입니다. 잘게 쪼갤수록 메모리는 여유롭지만 GPU 간 대화 비용이 늘어 느려질 수 있습니다. 그래서 어떤 축으로 어디까지 쪼갤지가 거대 학습 엔지니어링의 핵심 결정입니다. 한 가지 잊지 말아야 할 점은, 이 모든 것이 다음 토큰 예측이라는 단순한 목표(기초편 5단계)를 규모에서 실현하기 위한 받침대라는 것입니다. 목표는 그대로이고 규모만 커진 것입니다.

## 수식으로 보는 쪼갬의 효과

```
한 GPU 메모리 부담  ≈  (모델 + 기울기 + 옵티마이저 상태 + 활성값) / 나눈 정도
                         ↑ ZeRO·병렬이 이 분모를 키운다 (단, 통신 비용 ↑)
```

병렬화와 ZeRO는 한 GPU가 짊어질 부담을 여럿으로 나눠 분모를 키웁니다. 그래서 안 들어가던 모델이 들어갑니다. 대가는 GPU 간 통신입니다. 분모를 키울수록 대화가 늘어 효율이 떨어질 수 있습니다. 이 균형점을 찾는 것이 공학입니다.

## 흔한 오해

"GPU를 많이 붙이면 그만큼 빨라진다"는 오해입니다. 통신 비용 때문에 효율이 선형으로 늘지 않습니다(수확 체감). 무작정 쪼개면 오히려 느려질 수 있습니다.

"세 병렬 중 하나만 쓰면 된다"는 오해입니다. 대형 학습은 보통 셋을 조합(3D 병렬)하고 ZeRO를 더합니다. 각자 다른 자원 부족을 풀기 때문입니다.

"분산학습은 모델을 더 똑똑하게 만든다"는 오해입니다. 분산은 같은 학습을 가능하고 빠르게 하는 인프라일 뿐, 결과(모델 능력)를 바꾸지 않습니다. 능력은 데이터·규모·구조에서 옵니다.

## 정리

거대 모델은 한 GPU에 안 들어가 일을 여러 GPU에 쪼갭니다. 데이터 병렬(데이터 분담), 텐서 병렬(층 내부 분담), 파이프라인 병렬(층 분담), 그리고 중복을 없애는 ZeRO입니다. 핵심 트레이드오프는 통신 대 메모리이고, 실제로는 넷을 조합합니다. 목표(다음 토큰 예측)는 그대로 두고 규모만 떠받치는 인프라입니다. 다음 글에서는 그 규모를 어떻게 배분해야 가장 똑똑한 모델이 나오는지, Chinchilla의 교훈으로 넘어갑니다.
