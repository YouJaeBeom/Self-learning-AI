# 4. KV 캐시와 vLLM — 서빙 처리량 끌어올리기

지금까지는 모델 자체를 가볍게 했습니다(MoE·양자화·추측 디코딩). 마지막은 여러 사용자에게 동시에 서빙하는 시스템의 효율입니다. 열쇠는 어텐션이 만들어 내는 KV 캐시라는 메모리 덩어리를 어떻게 관리하느냐입니다. 그 답이 PagedAttention(vLLM)입니다. 흥미로운 점은, 모델을 전혀 바꾸지 않고 메모리 관리 방식만 바꿔 서빙 비용을 크게 낮췄다는 것입니다.

## 같은 GPU로 더 많은 요청을

LLM 서비스는 한 명이 아니라 수많은 사용자에게 동시에 답해야 합니다. 같은 GPU로 더 많은 요청을 처리할수록(처리량, throughput) 비용이 내려갑니다. 그런데 처리량을 막는 병목이 의외의 곳에 있습니다. 바로 KV 캐시 메모리입니다.

## KV 캐시란

기초편 4단계의 어텐션을 떠올려 보세요. 각 토큰은 K(키)와 V(값) 벡터를 만들어, 뒤에 오는 토큰들이 참고합니다. 생성은 한 토큰씩 진행되는데, 이전 토큰들의 K·V는 매번 똑같습니다. 그러니 매번 다시 계산하면 엄청난 낭비입니다. KV 캐시(KV cache)는 한 번 계산한 토큰들의 K·V를 저장해 두고 재사용하는 것입니다. 새 토큰은 자기 것만 계산하고 나머지는 캐시에서 꺼냅니다(S3에서 자세히 다뤘습니다).

효과는 생성 속도가 크게 빨라지는 것입니다(반복 계산 제거). 대가는 캐시가 메모리를 많이 먹는다는 것입니다. 문맥이 길수록, 사용자가 많을수록 KV 캐시가 GPU 메모리를 잡아먹습니다. 이 메모리가 동시 처리 가능한 요청 수를 제한합니다.

## PagedAttention (vLLM)

문제는 KV 캐시를 어떻게 메모리에 배치하느냐였습니다. 전통적 방식은 요청마다 연속된 큰 메모리 덩어리를 최대 길이로 미리 잡았습니다. 여기서 낭비가 폭발합니다. 답이 짧게 끝나면 잡아 둔 공간이 텅 빈 채 버려집니다(내부 단편화). 길이를 모르니 넉넉히 잡아야 해서 더 낭비됩니다. 남는 자투리 공간들이 흩어져 새 요청이 못 들어갑니다(외부 단편화).

PagedAttention은 운영체제의 가상 메모리와 페이징에서 아이디어를 빌립니다[1].

```
전통 방식: 요청마다 연속된 큰 방을 통째로 예약 → 빈자리 낭비

PagedAttention: KV 캐시를 작은 '페이지(블록)'로 잘라
                필요할 때마다 빈 페이지에 나눠 담음 (연속일 필요 없음)
                → 자투리 없이 빽빽하게, 메모리 거의 100% 활용
```

KV 캐시를 작은 고정 크기 블록으로 쪼개, 메모리 곳곳의 빈 블록에 흩어 담습니다(논리적으로는 이어진 것처럼). 효과는 메모리 낭비가 격감해 같은 GPU에 더 많은 요청을 동시에 올릴 수 있게 되고, 처리량이 급상승하는 것입니다. 보너스로 공통 접두사(같은 시스템 프롬프트 등)의 KV 블록을 여러 요청이 공유할 수도 있습니다. 이것이 vLLM이라는 서빙 엔진의 핵심입니다. vLLM이 서빙을 빠르게 하는 비결은 KV 캐시 메모리를 페이지로 잘게 관리해 낭비를 없앤 데 있습니다.

## 처리량과 메모리

```
동시 처리 가능 요청 수 ∝ 가용 메모리 ÷ 요청당 KV 캐시
```

KV 캐시 낭비를 줄이면 요청당 메모리가 줄어 동시 처리 요청 수가 늘고, 곧 처리량이 올라갑니다. PagedAttention은 분모(요청당 캐시 낭비)를 깎아 분자를 키운 셈입니다. 모델을 바꾸지 않고 메모리 관리만으로 비용을 낮춘 것입니다.

## 주차장 운영

KV 캐시는 주차장, 요청은 차입니다. 전통 방식은 차마다 대형 트럭용 자리를 통째로 예약하는 것입니다. 경차가 와도 큰 자리를 차지해 주차장이 금방 차고, 더 못 받습니다. PagedAttention은 주차장을 작은 칸으로 나눠, 차 크기만큼만 칸을 배정하는 것입니다. 빈틈없이 채워 훨씬 많은 차를 받습니다. 같은 주차장(GPU)으로 더 많은 차(요청)를 받는 것, 그게 처리량 향상입니다.

## 흔한 오해

"KV 캐시는 선택적 최적화다"라는 오해입니다. 없으면 매 토큰마다 전체 과거를 재계산해 생성이 비현실적으로 느려집니다. 사실상 필수입니다.

"PagedAttention은 모델을 바꾼다"는 오해입니다. 모델은 그대로 두고 메모리 관리 방식만 바꿉니다. 그래서 양자화·MoE와 함께 쓸 수 있습니다.

"처리량(throughput)과 지연(latency)은 같다"는 오해입니다. 처리량은 전체가 처리하는 양이고, 지연은 한 요청의 응답 속도입니다. PagedAttention은 주로 처리량을 키웁니다. 둘은 다른 목표입니다.

## 프리필과 디코드는 서로 다른 병목이다

LLM 서빙을 깊이 들여다보면, 생성이 성격이 완전히 다른 두 국면으로 나뉜다는 것을 알게 됩니다(S3에서 본 프리필과 디코드). 이 둘은 병목이 다르고, 그래서 최적화 방법도 다릅니다. 프리필은 입력 프롬프트 전체를 한꺼번에 처리하는 국면입니다. 토큰이 많아 연산량이 크고, GPU의 계산 능력을 꽉 채워 쓰는 연산 바운드입니다. 디코드는 답을 한 토큰씩 만드는 국면입니다. 매번 토큰 하나만 처리하니 연산량은 작지만, 그 토큰을 위해 거대한 모델 가중치와 KV 캐시를 메모리에서 읽어 와야 해서 메모리 대역폭 바운드입니다.

이 차이가 실무에 큰 함의를 갖습니다. 프리필은 연산이 병목이라 여러 요청을 묶어 처리(배칭)해도 한 요청당 속도가 크게 안 떨어지지만, 디코드는 메모리 읽기가 병목이라 배칭의 이득이 다른 방식으로 나타납니다. 그래서 최신 서빙 시스템은 프리필과 디코드를 분리해 각각에 맞게 스케줄링하거나, 두 국면을 잘게 섞어 GPU를 놀리지 않게 하는 등 정교한 기법을 씁니다. 사용자가 체감하는 첫 응답까지의 지연(프리필이 좌우)과 이후 토큰이 흐르는 속도(디코드가 좌우)가 다른 요인에 달려 있다는 점도 이 분리에서 나옵니다.

KV 캐시를 줄이는 또 다른 방향은 캐시 자체를 압축하거나 솎아 내는 것입니다. 모든 과거 토큰이 똑같이 중요하지는 않으므로, 어텐션에서 거의 주목받지 못하는 토큰의 K·V를 캐시에서 버려(eviction) 메모리를 아끼는 기법이 연구됩니다. GQA처럼 헤드 간 K·V를 공유하는 방법, 앞 글에서 본 KV 캐시 양자화, 그리고 이런 솎아 내기가 모두 같은 목표를 향합니다. 긴 문맥의 진짜 비용인 캐시 메모리를 줄여, 같은 하드웨어로 더 길고 더 많은 요청을 감당하는 것입니다.

## 정리

KV 캐시는 이전 토큰들의 K·V를 저장하고 재사용해 생성을 빠르게 하지만 메모리를 많이 먹습니다. PagedAttention(vLLM)은 그 캐시를 운영체제식 '페이지'로 잘게 관리해 메모리 낭비를 없애고, 같은 GPU로 더 많은 요청을 동시에 처리합니다(처리량 향상). 이로써 S7의 네 글 — MoE, 양자화, speculative decoding, KV 캐시·vLLM — 이 거대 모델을 싸고 빠르게 굴리는 한 그림으로 모였습니다. 다음 모듈에서는 모델의 지식 한계를 외부 검색으로 보완하는 RAG로 넘어갑니다.
