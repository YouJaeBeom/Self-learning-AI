# 1. ReAct와 도구 사용 — 추론과 행동을 엮기

기초편에서 에이전트의 심장을 "[생각 → 행동 → 관찰] 루프"라고 했습니다. 이 글은 그 루프의 정식 이름과 구조를 봅니다. 바로 ReAct입니다. 그리고 "행동"의 정체인 도구 호출(tool/function calling)이 어떻게 작동하는지 짚습니다. 핵심은 똑똑한 모델 하나가 아니라, 그 모델을 둘러싼 루프와 도구가 능력을 확장한다는 점입니다.

## 생각만으로는 부족하다

S6에서 CoT(생각의 사슬)로 추론을 길게 했습니다. 하지만 추론만으로는 한계가 있습니다. 모델은 최신 정보를 모르고, 정확한 계산에 약하고, 실제로 무언가를 실행하지 못합니다. CoT는 "머릿속 생각"만 늘릴 뿐, 바깥 세계와 상호작용하지 못합니다. 아무리 길게 생각해도 오늘 날씨를 알 수는 없고, 큰 수의 곱셈을 정확히 할 수도 없습니다.

여기서 발상이 나옵니다. 생각(reasoning)과 행동(acting)을 번갈아 엮자는 것입니다. 생각해서 무엇을 할지 정하고, 행동(도구)으로 정보를 얻고, 그걸 보고 다시 생각합니다.

## ReAct (Reasoning + Acting)

ReAct는 추론과 행동을 한 루프에서 교차시킵니다[1].

```
[Thought] 사용자가 내일 날씨를 묻는다. 나는 모른다 → 날씨 도구를 쓰자
[Action]  get_weather("서울", "내일")
[Observation] "비, 18도"                  ← 도구가 돌려준 결과
[Thought] 비가 오고 쌀쌀하다 → 이제 답할 수 있다
[Answer]  "내일은 비, 18도예요. 우산을 챙기세요."
```

Thought(생각)는 지금 무엇을 해야 하는지 추론하는 단계입니다. Action(행동)은 도구를 호출하는 단계입니다(검색·계산·코드 실행 등). Observation(관찰)은 도구의 결과를 받아 다음 생각의 입력으로 삼는 단계입니다. 이 셋을 여러 번 반복하며 복잡한 과제를 단계적으로 풉니다. CoT(생각만)에 행동과 관찰을 더한 것이 ReAct입니다. 추론만 하던 모델이 세계와 주고받으며 추론을 갱신하게 된 것입니다.

## 도구 호출(tool / function calling)

"Action"이 실제로 일어나는 방식이 도구 호출입니다.

```
① 모델에게 쓸 수 있는 도구 목록과 사용법을 알려 줌
   (예: get_weather(도시, 날짜), calculator(식), search(질의))
② 모델이 "지금은 get_weather를 써야겠다" 판단 →
   정해진 형식(예: JSON)으로 호출을 출력
③ 시스템이 그 호출을 실제로 실행 → 결과를 모델에 돌려줌
④ 모델이 결과를 보고 다음 생각/행동 또는 최종 답
```

핵심은 모델이 직접 실행하지 않는다는 것입니다. "이 도구를 이 인자로 불러 달라"는 요청을 생성할 뿐이고, 실제 실행은 바깥 시스템이 합니다. 그래서 모델의 약점(계산·최신성·실행)을 전문 도구가 메우고, 모델은 언제 무엇을 부를지 판단에 집중합니다. 더 나아가 모델이 스스로 언제 어떤 도구를 부를지를 학습할 수도 있습니다[2]. 이 분리는 보안과 신뢰성 측면에서도 중요합니다. 모델이 임의로 코드를 실행하는 게 아니라, 정해진 도구만 정해진 방식으로 호출하기 때문입니다.

## 도구를 든 탐정

탐정(에이전트)이 사건을 푸는 모습을 떠올려 보세요. 생각 단계에서는 "이 단서는 무엇을 뜻하지? 다음에 뭘 확인할까?"를 추론합니다. 행동 단계에서는 지문 감식을 의뢰하고(도구), 데이터베이스를 조회하고(도구), 현장을 다시 봅니다. 관찰 단계에서는 감식 결과를 받아 듭니다. 그리고 다시 생각합니다. "그렇다면 범인은…". 탐정은 모든 걸 머리로만 풀지 않습니다. 도구로 정보를 얻고, 그 결과로 추리를 갱신합니다. ReAct가 정확히 이 탐정의 일하는 방식입니다.

## 새 수식은 없다

ReAct와 도구 호출은 구조이지 수식이 아닙니다. 한 줄로 말하면 에이전트는 "모델 한 번 호출"을 [생각-행동-관찰]의 루프로 감싼 것입니다. 지능의 핵심은 모델에 있고, 능력의 확장은 루프와 도구에 있습니다. 이 관점이 에이전트를 이해하는 열쇠입니다. 더 똑똑한 에이전트는 더 큰 모델만이 아니라, 더 잘 설계된 루프와 더 적절한 도구에서도 나옵니다.

## 흔한 오해

"에이전트는 새로운 모델이다"라는 오해입니다. 보통 같은 LLM에 도구를 붙이고 [생각-행동-관찰] 루프로 감싼 시스템입니다. 핵심은 둘러싼 구조입니다.

"모델이 도구를 직접 실행한다"는 오해입니다. 모델은 호출 요청을 생성할 뿐, 실행은 바깥 시스템이 합니다. 보안과 신뢰성도 이 분리 덕분입니다.

"도구가 많을수록 똑똑하다"는 오해입니다. 도구가 많으면 언제 무엇을 쓸지 고르는 판단이 어려워져 실수가 늡니다. 적절한 도구 선택이 관건입니다.

## 도구 사용의 실패 모드와 신뢰성

에이전트가 도구를 쓴다고 해서 늘 똑똑하게 쓰는 것은 아닙니다. 실제로 배포하면 몇 가지 전형적인 실패가 나타납니다. 잘못된 도구를 고르거나(검색하면 될 일을 계산기로 시도), 도구는 맞는데 인자를 엉터리로 채우거나(없는 날짜·잘못된 형식), 도구가 에러를 돌려줬는데 그 에러를 무시하고 계속 진행하는 식입니다. 도구가 많아질수록 "언제 무엇을 어떻게 부를지"의 판단 공간이 넓어져 이런 실수가 늘어납니다. 그래서 도구를 무작정 많이 붙이기보다, 잘 정의된 소수의 도구와 명확한 사용 설명을 주는 편이 안정적입니다.

신뢰성을 높이는 핵심은 에러를 관찰의 일부로 다루는 것입니다. 도구가 실패하면 그 에러 메시지를 모델에게 관찰로 돌려주어, 모델이 "아, 인자가 틀렸구나" 하고 다음 행동에서 고치게 합니다. 사람이 명령어를 잘못 쳤을 때 에러를 보고 고치는 것과 같습니다. 여기에 더해, 모델이 임의의 위험한 동작을 하지 못하도록 도구의 권한을 제한하고(읽기만 허용 등), 각 호출을 기록해 무엇을 왜 했는지 추적할 수 있게 하는 것이 실무의 기본입니다.

도구 사용이 보편화되면서, 도구를 모델에 연결하는 방식을 표준화하려는 움직임도 있습니다(2026-06 기준). 도구마다 제각각이던 연결 규격을 통일하면, 한 번 만든 도구를 여러 에이전트가 재사용하고 생태계가 빠르게 자랍니다. 모델의 지능이 아니라 그 모델을 둘러싼 도구·에러 처리·권한·기록이라는 시스템이 에이전트의 실용성을 좌우한다는 점을, 이 표준화 흐름이 다시 보여 줍니다.

## 정리

ReAct는 [생각→행동→관찰]을 반복해 추론과 도구 사용을 엮습니다. CoT(생각만)에 행동·관찰을 더한 것입니다. 도구 호출에서 모델은 직접 실행하지 않고 호출 요청만 생성하고, 실제 실행은 바깥 시스템이 맡아 모델의 약점(계산·최신성)을 메웁니다. 다음 글에서는 이 루프를 더 긴 과제로 데려가는 계획·메모리·반성으로 넘어갑니다.
