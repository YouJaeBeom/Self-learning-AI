# 3. 멀티에이전트 — 여럿이 나눠 더 잘하기

지금까지는 한 에이전트가 도구를 쓰고 계획하고 반성했습니다. 한 걸음 더 가면, 여러 에이전트가 역할을 나눠 협업합니다. 한 명이 모든 역할을 떠맡을 때 생기는 과부하와 맹점을, 분업과 상호 점검으로 푸는 것입니다. 다만 에이전트를 늘리는 것이 늘 답은 아니며, 분업이 명확할 때만 값한다는 점이 중요합니다.

## 한 명이 다 하기엔 벅차다

복잡한 일을 한 에이전트가 처음부터 끝까지 하면 두 문제가 생깁니다. 하나는 맥락 과부하입니다. 한 컨텍스트에 모든 역할(계획·실행·검증)을 욱여넣으면 헷갈리고 품질이 떨어집니다. 다른 하나는 단일 관점입니다. 자기 답을 자기가 검토하면 맹점을 못 봅니다(기초편 6단계의 "달변의 신입사원" 비유). 사람도 자기 글의 오타를 잘 못 찾는 것과 같습니다.

여기서 발상이 나옵니다. 역할을 나눠 여러 에이전트가 협업하자는 것입니다. 각자 좁은 임무에 집중하고, 서로 점검하게 합니다.

## 협업의 두 형태

첫째는 역할 분담(role specialization)입니다. 에이전트마다 다른 역할을 줍니다. 예를 들어 기획자 → 작성자 → 검토자 → 편집자로 이어지는 파이프라인입니다. 각자 좁은 임무라 깊이가 있고, 결과를 이어받아 완성도를 높입니다. 소프트웨어 개발에 빗대면 설계자, 구현자, 테스터, 리뷰어가 한 팀을 이루는 것과 같습니다. 이런 역할 기반 협업 프레임워크가 여럿 제안됐습니다[3].

둘째는 토론·비평(debate / critique)입니다. 여러 에이전트가 같은 문제를 따로 풀고, 서로의 답을 비평하거나 투표해 더 나은 결론에 이릅니다. S6의 self-consistency(여러 추론 다수결)를 역할 있는 에이전트로 키운 셈입니다. 한 에이전트가 생성하고 다른 에이전트가 반박·검증하는 구도는 맹점을 줄입니다. 여러 에이전트의 토론이 사실성과 추론을 개선한다는 것이 연구로 확인됐습니다[2].

## 언제 멀티에이전트가 값하나

멀티에이전트는 공짜가 아닙니다. 에이전트가 늘면 비용·지연·복잡도가 함께 늡니다. 그래서 값하는 경우를 가립니다.

| 상황 | 멀티에이전트가 값하나 |
|---|---|
| 명확히 나눌 수 있는 하위 작업 | 값함(역할 분담) |
| 교차 검증이 중요 | 값함(비평·토론) |
| 넓게 탐색해야 함 | 값함(병렬) |
| 단순한 과제 | 과함(조율 비용만 늘어남) |

핵심 설계 난제는 조율(orchestration)입니다. 누가 무엇을 먼저 하고, 결과를 어떻게 합치고, 충돌하면 누가 결정하는가입니다. 오케스트레이터 역할의 에이전트가 이를 관장합니다. 분업의 이득이 조율의 비용을 넘어설 때 비로소 멀티에이전트가 의미를 가집니다.

## 새 수식은 없다

멀티에이전트도 구조입니다. 한 줄로 말하면 여러 모델 호출을 역할과 흐름으로 엮은 것입니다. 이득(분업·교차검증)과 비용(조율·지연)의 거래를 설계하는 일입니다. 그래서 멀티에이전트 설계는 알고리즘 문제라기보다 일종의 조직 설계 문제에 가깝습니다.

## 신문 편집국

신문 한 부는 한 사람이 만들지 않습니다. 기자가 취재해 쓰고(작성자), 데스크가 사실과 논조를 점검하고(검토자), 편집자가 지면을 통일합니다(편집자). 각자 좁은 전문에 집중하고 서로 점검하기에, 한 사람이 다 하는 것보다 정확하고 일관됩니다. 멀티에이전트도 똑같은 원리입니다. 역할을 나누고 서로 검증하는 구조가 단일 에이전트의 과부하와 맹점을 줄입니다. 다만 작은 기사 하나에 편집국 전체를 동원하지 않듯, 과제 규모에 맞춰 협업의 규모도 정해야 합니다.

## 흔한 오해

"에이전트가 많을수록 좋다"는 오해입니다. 조율 비용과 지연이 늘어, 단순 과제는 한 에이전트가 낫습니다. 분업이 명확할 때 값합니다.

"멀티에이전트는 여러 모델이 필요하다"는 오해입니다. 같은 모델을 다른 역할 프롬프트로 여러 번 부르는 경우가 많습니다. 핵심은 모델 수가 아니라 역할 분담입니다.

"토론하면 항상 정답에 수렴한다"는 오해입니다. 모두 같은 방식으로 틀리면(S6 self-consistency의 한계와 같은 뿌리) 토론도 틀립니다. 다양성이 있어야 교차검증이 듣습니다.

## 조율의 패턴들

멀티에이전트를 실제로 짤 때 떠오르는 질문은 "어떻게 연결하느냐"입니다. 몇 가지 전형적인 패턴이 있습니다. 첫째는 오케스트레이터-워커 패턴입니다. 한 관리자 에이전트가 일을 잘게 나눠 여러 작업자 에이전트에게 분배하고, 결과를 모아 종합합니다. 작업을 명확히 쪼갤 수 있을 때 효과적이고, 책임 소재가 분명합니다. 둘째는 순차 파이프라인입니다. 기획 → 작성 → 검토 → 편집처럼 각 단계를 다른 에이전트가 맡아 결과를 이어받습니다. 단계가 분명한 작업에 맞습니다. 셋째는 토론 패턴입니다. 여러 에이전트가 같은 문제를 따로 풀고 서로 비평해 더 나은 결론에 이릅니다. 한 관점의 맹점을 줄이는 데 좋습니다.

어떤 패턴이든 공통으로 부딪히는 어려움이 있습니다. 에이전트 사이의 의사소통입니다. 한 에이전트의 출력이 다른 에이전트의 입력이 되는데, 그 사이에서 정보가 왜곡되거나 누락되면 오류가 누적됩니다. 또 한 에이전트가 틀리면 그 틀린 결과가 다음 에이전트로 전파됩니다. 그래서 검증하는 에이전트를 따로 두거나, 중간 결과를 구조화된 형식으로 주고받게 해 왜곡을 줄입니다.

가장 중요한 판단은 "이 일에 멀티에이전트가 정말 필요한가"입니다. 에이전트를 늘리면 비용과 지연이 비례해 늘고, 조율이 복잡해집니다. 단순한 작업을 여러 에이전트로 나누면 조율 비용만 추가될 뿐 이득이 없습니다. 분업이 명확하고, 교차 검증이 품질을 크게 올리고, 병렬 탐색이 필요한 경우에만 그 추가 비용이 정당화됩니다. "더 많은 에이전트"가 아니라 "꼭 필요한 만큼의 분업"이 좋은 설계입니다.

## 정리

멀티에이전트는 역할을 나눠(기획·작성·검토) 협업하거나 서로 비평·투표해 한 에이전트의 과부하와 맹점을 줄입니다. 단, 조율 비용이 있어 분업이 명확하고 교차검증이 중요할 때 값합니다. 빠르게 발전 중인 영역이라(2026-06 기준) 표준 정설보다 대표적 접근들로 이해하는 게 좋습니다. 다음 글에서는 이렇게 만든 에이전트와 모델이 정말 더 나은지를 어떻게 재는가, 곧 LLM 평가로 모듈을 닫습니다.
