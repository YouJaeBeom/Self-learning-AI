# 2. RLHF 파이프라인 — 보상 − KL을 최대화하기

앞 글에서 SFT 모델과 보상 모델이 준비됐습니다. 이제 세 번째 단계인 강화학습입니다. 모델이 답을 만들면 보상 모델이 점수를 매기고, 그 점수가 높아지도록 모델을 밉니다. 단순해 보이지만, 여기에는 고삐 하나가 결정적입니다. 바로 KL 페널티입니다. 이 글은 RLHF가 최대화하는 `보상 − KL`이라는 목적함수가 무엇을 뜻하는지, 그리고 왜 그 고삐가 없으면 모델이 무너지는지를 다룹니다.

## 점수만 좇으면 망가진다

보상 모델이라는 채점기가 생겼으니, 단순하게 생각하면 "점수를 최대로 올리자"입니다. 그런데 여기 함정이 있습니다. 보상 모델은 앞 글에서 봤듯 불완전한 근사입니다. 사람 취향을 흉내 냈을 뿐 완벽하지 않습니다. 점수만 맹목적으로 올리면 모델은 채점기의 허점을 파고드는 이상한 답으로 흘러갑니다. 이것이 보상 해킹(reward hacking)입니다. 예를 들어 무조건 길게 쓰거나, 채점기가 좋아하는 특정 문구를 반복하거나, 형식만 그럴듯하게 꾸미는 식입니다. 점수는 높은데 사람이 보기엔 엉망인 답이 나옵니다.

그래서 RLHF의 목표는 두 개입니다. 점수는 높이되, 원래(SFT) 모델에서 너무 멀어지지는 말라는 것입니다. 이 두 번째 조건이 보상 해킹을 막는 핵심입니다.

## 정책·보상·KL

강화학습 용어를 LLM에 대응시켜 잡아 보겠습니다. 정책(policy)은 우리가 학습하는 LLM입니다. 상황(질문)에서 행동(다음 토큰)을 고르는 전략입니다. 보상(reward)은 보상 모델이 매긴 답의 점수입니다. 기준 정책(reference policy)은 학습 시작점인 SFT 모델로, "원래의 나"에 해당합니다. KL 페널티는 현재 정책이 기준 정책에서 얼마나 벗어났는지를 재는 벌점입니다.

```
질문 → [현재 LLM(정책)] → 답 생성
                  │
                  ▼
        [보상 모델] → 점수 r
                  │
   점수는 높이고, SFT에서 벗어난 정도(KL)는 누르도록
                  ▼
        정책(LLM) 업데이트  (PPO로 조금씩)
                  └────────── 반복 ──────────┘
```

## 보상 − KL 목적함수

RLHF가 최대화하는 것을 한 줄로 적으면(의미만) 이렇습니다.

```
maximize  E[ r(x, y) ]  −  β · KL( π(현재) ‖ π(기준=SFT) )
```

기호를 풀면, `r(x, y)`는 질문 x에 모델이 낸 답 y의 보상(점수)입니다. 이걸 높이고 싶습니다. `KL(π현재 ‖ π기준)`은 현재 모델의 답 분포가 SFT 모델에서 벗어난 거리입니다. 이건 작게 유지하고 싶습니다. `β`(베타)는 둘의 저울 손잡이입니다. 크면 "원래에서 벗어나지 마"를 강하게, 작으면 "점수에 더 욕심내"를 뜻합니다.

이 식이 말하는 것은 "보상 모델이 좋아할 답을 하되, 원래 말투와 능력을 잃을 만큼 변하지는 마라"입니다. KL 항이 없으면 모델은 채점기의 허점만 노리다 언어 능력 자체가 무너집니다. KL은 그 폭주를 막는 안전벨트입니다. 이 단계의 실제 최적화는 보통 PPO(Proximal Policy Optimization)라는 강화학습 알고리즘으로 합니다[2]. PPO의 핵심 정신도 비슷합니다. 한 번에 너무 크게 바꾸지 말고 조금씩 갱신하라는 것입니다. 유도와 세부는 원문으로 넘깁니다.

## 전체 파이프라인 한눈에

기초편의 3단계를 데이터 흐름까지 더해 다시 그리면 이렇습니다.

```
① SFT        : (지시, 모범답안)으로 기본 대화법 모방
        ▼
② 보상 모델   : (답 A ≻ 답 B) 선호 데이터로 채점기 학습
        ▼
③ RLHF(PPO)  : LLM이 답 생성 → 보상 모델 채점 → (보상 − βKL) 최대화로 LLM 갱신 → 반복
```

이 셋이 모여 "베이스 모델에서 도움 되고 안전한 대화 모델"로의 변신을 완성합니다[1]. 다만 세 번째 단계는 복잡하고 불안정합니다. 정책·보상·기준이라는 세 모델을 동시에 다뤄야 하고, PPO는 하이퍼파라미터에 민감합니다. 이 복잡함을 줄이려는 시도가 다음 글의 DPO입니다.

## 코치와 함께 고쳐 쓰기, 단 본색은 유지

학생(LLM)이 답을 씁니다. 코치(보상 모델)가 점수를 줍니다. 학생은 점수를 높이려 답을 고쳐 갑니다. 여기에 한 가지 규칙이 붙습니다. "네 원래 문체(SFT)에서 너무 벗어나면 감점"입니다. 이것이 KL입니다. 이 규칙이 없으면 학생은 코치의 취향을 악용한 괴상한 글쓰기로 빠져, 점수는 높은데 사람이 보기엔 엉망인 답을 내놓습니다. 좋은 정렬은 능력을 끌어올리면서도 본색을 잃지 않는 균형에서 나옵니다.

## 흔한 오해

"KL 페널티는 성능을 깎는 군더더기다"라는 오해입니다. KL이 없으면 보상 해킹으로 모델이 무너집니다. 정렬을 가능케 하는 핵심 안전장치입니다.

"RLHF는 모델에 새 지식을 넣는다"는 오해입니다. 주로 이미 가진 능력을 끌어내고 다듬을 뿐입니다. 지식은 사전학습에서 옵니다. RLHF는 모델의 행동 방식을 조정하는 것에 가깝습니다.

"RLHF는 곧 PPO다"라는 오해입니다. RLHF는 사람 피드백으로 정렬한다는 큰 틀이고, PPO는 그 안에서 쓰는 한 가지 최적화 도구입니다. DPO·GRPO 등 다른 길도 있습니다.

## PPO가 동시에 굴리는 네 모델

RLHF의 세 번째 단계가 왜 그렇게 무겁고 불안정한지는, PPO가 실제로 몇 개의 모델을 동시에 메모리에 올려 두고 돌리는지를 보면 분명해집니다. 보통 네 개입니다. 정책(학습 중인 LLM), 기준 정책(SFT 모델, KL을 재는 기준), 보상 모델(답을 채점), 그리고 가치 함수(critic, 각 상태가 앞으로 얼마나 좋은 보상을 받을지 추정)입니다. 거대 모델 네 개를 한꺼번에 다루니 메모리와 연산이 폭발하고, 이 모델들이 서로 영향을 주고받으며 학습이 미묘하게 흔들립니다.

특히 가치 함수가 골칫거리입니다. 보상은 보통 답 전체가 끝난 뒤 한 번 주어지는데(예: "이 답은 7점"), 강화학습은 그 보상을 각 토큰의 기여로 나눠 배분해야 합니다. 이 배분을 돕는 것이 가치 함수이고, 그것이 부정확하면 학습 신호가 잡음투성이가 됩니다. 그래서 PPO 구현에는 보상 정규화, 어드밴티지 추정(GAE), 클리핑 범위 조정 같은 세심한 튜닝이 필요하고, 하나만 어긋나도 발산하기 쉽습니다.

이 복잡함을 이해하면 왜 후학습 연구가 두 방향으로 갈라졌는지가 보입니다. 한쪽은 이 무거운 PPO를 통째로 걷어내려는 시도(다음 글의 DPO)이고, 다른 한쪽은 가장 무거운 부품인 가치 함수만 빼서 가볍게 만드는 시도(S6에서 다시 만날 GRPO)입니다. 둘 다 "RLHF는 강력하지만 너무 비싸고 까다롭다"는 같은 문제의식에서 출발합니다.

## 정리

RLHF의 세 번째 단계는 `보상 − β·KL`을 최대화합니다. 보상 모델 점수는 높이되 SFT에서 멀어진 정도(KL)는 누릅니다. KL은 보상 해킹을 막는 안전벨트이고, 최적화는 보통 PPO로 합니다. 강력하지만 세 모델을 동시에 다뤄 복잡하고 불안정합니다. 다음 글에서는 이 복잡함을 통째로 걷어내려는 DPO로 넘어갑니다.
