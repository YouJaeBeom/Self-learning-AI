# 2. FedAvg 직관 — 각자 배우고 가중평균으로 합치기

1장에서 "모델 업데이트만 모아 합친다"고 했습니다. 그런데 그 *합치는 방법*은 한 가지가 아닙니다. 매 스텝마다 합칠 수도 있고, 한참 배운 뒤 합칠 수도 있고, 합칠 때 무게를 어떻게 줄지도 정해야 합니다. 이 모든 선택지 중에서 가장 단순하면서도 가장 강력해, 연합학습을 세상에 알린 알고리즘이 **FedAvg(Federated Averaging)**입니다 [1].

FedAvg는 이름 그대로 "연합(Federated) 평균내기(Averaging)"가 전부입니다. 하지만 *언제* 평균내고 *어떻게* 무게를 주느냐에 모든 묘미가 숨어 있습니다. 이 글은 그 두 가지 — "여러 스텝 뒤에" 그리고 "데이터 양으로 가중해서" — 를 왜 그렇게 하는지 봅니다.

---

## 목적 — "합치기"를 가장 단순하게

가장 순진한 방법부터 생각해 봅시다. 라운드마다 클라이언트가 *딱 한 스텝(미니배치 하나)*만 학습하고, 곧바로 서버가 그 결과를 평균낸다면 어떨까요? 이건 사실 1단계에서 배운 평범한 SGD를 그냥 여러 기기에 흩뿌린 것(분산 SGD)과 똑같습니다. 잘 수렴하긴 합니다.

문제는 **통신**입니다. 매 스텝마다 모델 전체를 내려보내고 다시 올려받아야 하니, 통신이 미친 듯이 잦습니다. 모델이 수백 MB인데 그걸 매 스텝마다 수백만 대의 휴대폰과 주고받는다고 상상해 보세요. 휴대폰 배터리도, 통신망도, 서버도 버티지 못합니다. 연합학습에서는 *계산보다 통신이 훨씬 비싼* 병목이거든요.

> FedAvg의 핵심 발상: **클라이언트가 한 스텝이 아니라 *여러 스텝(여러 에폭)* 충분히 로컬 학습을 한 뒤, 그 결과를 한 번에 가중평균하자.** 통신 횟수를 수십 배 줄이면서도, 놀랍게도 잘 수렴합니다.

이 "여러 스텝 뒤에 합친다"가 FedAvg를 분산 SGD와 가르는 결정적 차이입니다. 클라이언트가 로컬에서 더 많이 계산할수록 서버와 덜 자주 통신해도 되고, 통신이 병목인 연합학습에서 이건 곧 속도와 비용의 승리입니다.

---

## 핵심 개념 — 가중평균이 핵심

서버가 클라이언트들의 모델을 합칠 때, 단순한 산술평균이 아니라 **데이터 양으로 가중평균**합니다. 이 차이가 생각보다 큽니다.

> 데이터가 많은 클라이언트의 모델에 *더 큰 무게*를 줍니다. 1만 건으로 꼼꼼히 배운 모델과 100건으로 대충 배운 모델을 똑같이 1:1로 평균낸다면, 신뢰도가 100배 낮은 모델이 절반의 발언권을 갖게 되니 불공정하고, 결과도 나빠집니다.

수식으로는 이렇게 씁니다.

```
글로벌 모델 w = Σ_k (n_k / n) × w_k

   w_k = 클라이언트 k가 로컬 학습으로 얻은 모델
   n_k = 클라이언트 k의 데이터 수
   n   = 전체 데이터 수 (= Σ_k n_k)
```

가중치 `n_k / n`은 그 클라이언트가 전체 데이터에서 차지하는 비율이고, 그 비율이 곧 **글로벌 모델에서의 발언권**입니다. 데이터를 많이 가진 클라이언트일수록 결과를 더 많이 끌어당깁니다. 모든 가중치 `n_k/n`을 더하면 정확히 1이 되므로(`Σ n_k/n = n/n = 1`), 이 식은 어김없는 가중평균입니다.

작은 예로 감을 잡아 봅시다. 클라이언트 A(데이터 900건)가 어떤 가중치를 `2.0`으로, 클라이언트 B(데이터 100건)가 같은 가중치를 `4.0`으로 배웠다면,

- 그냥 산술평균: `(2.0 + 4.0) / 2 = 3.0`
- 데이터 양 가중평균: `0.9 × 2.0 + 0.1 × 4.0 = 1.8 + 0.4 = 2.2`

데이터가 9배 많은 A 쪽으로 결과가 끌려가는 게 보입니다. 소수 데이터로 배운 B의 출렁임에 글로벌 모델이 휘둘리지 않게 막아 주는 거죠.

## 직관 — 반 평균을 낼 때 인원수로 가중

두 학급의 평균 점수를 합쳐 학년 평균을 내는 상황을 떠올려 보세요. A반(40명) 평균이 70점, B반(10명) 평균이 90점입니다.

- 그냥 `(70 + 90) / 2 = 80`이라고 하면 틀립니다. 10명짜리 작은 반이 큰 반과 똑같은 비중을 차지해 *과대 반영*되니까요.
- 옳은 답은 인원수로 가중한 `(40×70 + 10×90) / 50 = (2800 + 900) / 50 = 74`점입니다.

FedAvg의 가중평균이 정확히 이것입니다. **데이터가 많은 클라이언트일수록 글로벌 모델에 더 크게 기여**하도록, 데이터 수로 무게를 줍니다. "사람 수가 많은 반이 학년 평균을 더 많이 좌우한다"와 "데이터가 많은 클라이언트가 글로벌 모델을 더 많이 좌우한다"는 완전히 같은 원리입니다. 산술평균을 쓰면 데이터 10건짜리 클라이언트와 100만 건짜리 클라이언트가 똑같은 한 표를 행사하는, 말이 안 되는 일이 벌어집니다.

## 방법론 — FedAvg 한 라운드

FedAvg 한 라운드의 전체 흐름을 단계별로 펼치면 이렇습니다. 1장의 일반적인 라운드에 "여러 스텝"과 "가중평균"이 구체적으로 들어찬 모습입니다.

```
① 서버가 글로벌 모델 w를 (전체가 아니라 일부) 클라이언트에게 배포
        │
        ▼
② 각 클라이언트 k:
     자기 데이터로 *여러 스텝/에폭(E)* 동안 SGD를 돌려 w_k를 만든다
     (← 핵심: 한 스텝이 아니라 푹 익힘)
        │
        ▼
③ 클라이언트들이 w_k (또는 변화량 Δw_k = w_k − w)를 서버로 전송
     (← 데이터는 안 보냄, 가중치만)
        │
        ▼
④ 서버: w ← Σ_k (n_k / n) w_k   (데이터 양 가중평균)
        │
        ▼
⑤ 수렴할 때까지 ①~④ 반복
```

이 흐름을 시간 순으로 펼쳐 보면, 클라이언트들이 각자 다른 출발점이 아니라 *같은 글로벌 모델에서 출발*해 잠시 갈라졌다가 다시 평균으로 모이는 모습이 보입니다.

```
라운드 t 시작:  모두 같은 w 에서 출발
                       w
                    ╱  │  ╲
          (로컬학습) ╱    │    ╲ (로컬학습)
              w_A      w_B      w_C     ← 각자 데이터로 여러 스텝, 잠시 갈라짐
                    ╲    │    ╱
              (가중평균) ╲  │  ╱
                       w'              ← 다시 하나로 모임 (라운드 t+1의 출발점)
```

여기서 손에 쥐고 흔들 수 있는 **손잡이(하이퍼파라미터) 둘**이 결과를 좌우합니다.

- **로컬 에폭 수(E)**: 한 라운드에서 클라이언트가 자기 데이터를 몇 번이나 훑을지입니다. `E`가 크면 통신은 줄지만(좋음), 클라이언트들이 *각자 자기 데이터에만 너무 깊이 맞춰져 서로 다른 방향으로 멀리 가버리는* 위험이 커집니다. 이를 **client drift(클라이언트 표류)**라 부릅니다. 위 그림에서 `w_A`, `w_B`, `w_C`가 서로 너무 멀리 벌어지면, 그 평균 `w'`가 어느 누구에게도 좋지 않은 어정쩡한 지점이 될 수 있죠.
- **참여 클라이언트 비율(C)**: 매 라운드 모든 클라이언트가 참여하는 게 아니라 *일부만* 뽑아 참여시킵니다. 특히 교차 기기에서는 휴대폰이 *충전 중이고 와이파이에 연결됐고 화면이 꺼져 있을 때만* 참여하는 식이라, 매 라운드 극히 일부만 동원됩니다. 더 많이 뽑으면 한 라운드가 더 안정적이지만 그만큼 통신·대기 비용이 늘어납니다.

| 손잡이 | 키우면(↑) | 줄이면(↓) |
|---|---|---|
| 로컬 에폭 E | 통신 ↓ (좋음), client drift ↑ (위험) | 통신 ↑ (비쌈), 안정적 |
| 참여 비율 C | 라운드당 안정성 ↑, 통신·대기 ↑ | 빠르고 쌈, 라운드 결과 출렁임 |

> FedAvg를 잘 쓴다는 건 결국 이 두 손잡이로 **"통신을 줄이려는 욕심"과 "client drift로 인한 불안정"** 사이의 균형점을 찾는 일입니다. `E`를 무작정 키워 통신을 아끼려다 표류가 심해져 오히려 수렴이 망가지는 게 가장 흔한 실패입니다.

## 수학 — 로컬 SGD + 가중평균

조금 더 또렷이 적어 봅시다. FedAvg는 두 부분의 반복입니다.

- **로컬 학습**: 각 클라이언트가 자기 데이터로 SGD를 *여러 스텝* 돌립니다. 한 스텝은 우리가 1단계에서 배운 바로 그 `w ← w − η ∇L_k(w)`이고, 이걸 E 에폭만큼 반복해 `w_k`를 얻습니다. 클라이언트마다 데이터가 다르니 손실 함수 `L_k`도 다르고, 따라서 도착점 `w_k`도 제각각입니다.
- **집계**: 서버가 `w ← Σ_k (n_k/n) w_k`로 데이터 양 가중평균을 냅니다.

`E = 1`(한 스텝만 학습)이고 모든 클라이언트가 참여하면, FedAvg는 사실 분산 SGD와 정확히 같아집니다. FedAvg의 모든 이득은 `E > 1`, 즉 *로컬에서 푹 익힌 뒤 가끔 합치는* 데서 나옵니다.

> 한 줄: **"로컬에서 여러 스텝 → 데이터 양으로 가중평균"의 반복.** 이 단순한 두 동작의 반복이 데이터를 한곳에 모으지 않고도 글로벌 모델을 키웁니다. 단순함이 곧 FedAvg가 사실상 표준이 된 이유입니다.

## 한 라운드를 숫자로 따라가기

추상적인 식만 보면 감이 안 잡힐 수 있으니, 아주 작은 가상의 라운드를 끝까지 따라가 봅시다. 클라이언트가 셋이고, 모델은 가중치가 딱 하나(`w`)인 장난감 모델이라고 합시다. 라운드 시작 시 글로벌 모델은 `w = 1.00`입니다.

```
라운드 시작:  글로벌 모델 w = 1.00 을 세 클라이언트에 배포

  클라이언트 A (데이터 800건): 로컬 학습 후 w_A = 1.20
  클라이언트 B (데이터 150건): 로컬 학습 후 w_B = 0.60
  클라이언트 C (데이터  50건): 로컬 학습 후 w_C = 1.80
       └─ 전체 데이터 n = 800 + 150 + 50 = 1000건
```

각자 자기 데이터로 여러 스텝 학습했더니 서로 다른 곳에 도착했습니다(데이터가 다르니 당연합니다). 이제 서버가 합칩니다.

- **만약 산술평균이라면**: `(1.20 + 0.60 + 1.80) / 3 = 1.20`. 데이터 50건짜리 C의 큰 출렁임(1.80)이 800건짜리 A와 똑같은 1/3 발언권을 차지해, 결과가 위로 끌려갑니다.
- **데이터 양 가중평균(FedAvg)**:
  `w = (800/1000)·1.20 + (150/1000)·0.60 + (50/1000)·1.80`
  `  = 0.8·1.20 + 0.15·0.60 + 0.05·1.80`
  `  = 0.960 + 0.090 + 0.090 = 1.14`

데이터가 압도적으로 많은 A 쪽으로 결과가 끌리고, 데이터가 적은 C의 과격한 움직임(1.80)은 `0.05`라는 작은 무게로 눌려 거의 반영되지 않았습니다. 새 글로벌 모델 `w = 1.14`가 다음 라운드의 출발점이 되고, 이 과정을 수십~수백 번 반복하며 모델이 자라납니다.

> 이 작은 예에서 보이듯, 가중평균은 *신뢰할 만한(데이터 많은) 클라이언트의 의견을 더 듣고, 표본이 빈약한 클라이언트의 변덕은 눌러* 글로벌 모델을 안정시킵니다.

## 흔한 오해

- **"FedAvg는 항상 잘 수렴한다"가 아닙니다.** 클라이언트들의 데이터 분포가 서로 많이 다르면(non-IID), 각 클라이언트가 *자기 데이터에 맞는 서로 다른 최적점*으로 끌고 가 client drift가 심해지고, 그 평균이 어디로도 잘 수렴하지 않을 수 있습니다. 이를 보완하려고 로컬 학습이 글로벌에서 너무 멀어지지 않게 묶는 FedProx 등이 제안됐습니다(심화편·논문편).
- **"로컬 에폭 E는 클수록 좋다"가 아닙니다.** E를 키우면 통신은 줄지만 client drift가 커져 *오히려 전체 성능이 나빠질 수 있습니다.* 통신 절약과 안정성 사이의 균형이 필요합니다. 공짜 점심은 없습니다.
- **"가중평균은 그냥 평균과 비슷하다"가 아닙니다.** 데이터 양 가중이 핵심입니다. 이걸 빼고 산술평균을 쓰면, 데이터가 적은 클라이언트가 과대 반영되어 글로벌 모델이 소수 의견에 휘둘립니다. `n_k/n`이라는 무게가 FedAvg의 공정성을 떠받칩니다.

---

## 한 줄 요약

> **FedAvg는 '클라이언트가 여러 스텝 로컬 학습 → 서버가 데이터 양으로 가중평균'을 반복한다 — 통신을 확 줄이면서, 데이터를 한곳에 모으지 않고도 글로벌 모델을 키우는 연합학습의 기본 알고리즘이다.**
> 단, 클라이언트 분포가 다르고(non-IID) 로컬 에폭이 과하면 client drift로 흔들린다. 통신 절약과 안정성의 균형이 전부다.
