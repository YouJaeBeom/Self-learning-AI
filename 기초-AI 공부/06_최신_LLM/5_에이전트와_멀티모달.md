# 5. 에이전트와 멀티모달 — 행동하고, 보고 듣기

지금까지의 LLM은 *글을 받아 글을 내놓는* 상자였습니다. 똑똑한 상자지만, 어디까지나 텍스트 안에 갇혀 있었죠. 이 글은 그 상자가 두 방향으로 벽을 허무는 이야기입니다. **① 도구를 써서 '행동'하는 에이전트**, 그리고 **② 글을 넘어 이미지·소리까지 다루는 멀티모달.** 앞 글까지의 모든 개념이 여기서 *실제로 일하는 시스템*으로 모입니다.

---

## 목적 — 말만 하는 모델의 한계를 넘기

LLM은 똑똑하지만, 혼자서는 못 하는 일이 의외로 많습니다.

- **최신 정보**를 모릅니다 (학습 이후의 일은 모름 → 검색이 필요).
- **정확한 계산**에 약합니다 (큰 수의 곱셈에서 자주 틀림 → 계산기가 필요).
- 실제로 **무언가를 실행**하지 못합니다 (이메일 보내기, 코드 실행, 예약 등 → 외부 도구가 필요).
- 글만 압니다 (사진·그래프·음성은 어떻게? → 감각이 필요).

흥미롭게도, 이 약점 중 일부는 *모델을 더 키워도* 잘 해결되지 않습니다. 아무리 큰 모델도 "오늘 환율"은 학습에 없으니 알 수 없고, 계산은 본질적으로 도구가 더 정확합니다. 5단계의 스케일링 법칙으로 풀리는 한계(지식·유창성)와, *스케일로는 안 풀리는 한계*(최신성·정확한 연산·실행력)가 따로 있는 셈입니다. 후자를 메우는 길이 바로 이 글의 두 확장입니다.

> 그래서 두 방향으로 확장합니다 — 모델에게 **손발(도구)**을 달아 주고(에이전트), **눈과 귀(이미지·음성)**를 달아 줍니다(멀티모달). 모델 자체를 바꾸기보다, 모델을 *둘러싸는 구조*를 키우는 접근입니다.

---

## 핵심 개념 — LLM 에이전트와 도구 사용

**에이전트(agent)**는 LLM이 *생각만 하는 게 아니라*, **도구(tool)를 불러 행동하고, 그 결과를 보고 다음 행동을 정하는** 구조입니다. 핵심은 **도구 사용(tool use / function calling)**입니다. 모델이 "지금은 내가 직접 답할 게 아니라 *검색 도구를 써야겠다*"고 스스로 판단하면, 정해진 형식으로 도구를 호출하고 결과를 받아 이어 갑니다.

```
사용자: "내일 서울 날씨에 맞는 옷차림 추천해줘"
   │
   ▼
[생각]  날씨를 모른다 → 날씨 도구를 써야겠다
   │
   ▼  [행동]  get_weather("서울", "내일")  호출
   │
   ▼  [관찰]  결과 수신: "비, 18도, 습함"
   │
   ▼
[생각]  비가 오고 쌀쌀하다 → 이제 답할 수 있다
   │
   ▼
답: "내일 서울은 비가 오고 18도예요. 방수 외투와 우산을 챙기세요."
```

### 생각-행동-관찰 루프

이 **[생각(Reason) → 행동(Act, 도구) → 관찰(Observe) → 다시 생각]** 순환이 에이전트의 심장입니다. 2번 글의 *생각의 사슬(추론)*에 **행동**이 결합된 형태죠. 이 둘을 명시적으로 엮은 대표 방법이 **ReAct**입니다 [1] — 추론 흔적(reasoning trace)이 행동 계획을 세우고 갱신하게 하고, 행동이 외부 세계에서 새 정보를 가져오게 해, *추론과 행동이 서로를 돕게* 만듭니다.

루프가 한 번으로 끝나지 않는다는 점이 중요합니다. 복잡한 일은 여러 번의 [생각-행동-관찰]을 거쳐 *스스로 계획하고 수행*합니다. 예컨대 "이번 주 가장 싼 항공권을 찾아 예약 링크를 정리해 줘"라면, 검색 → 날짜별 비교 → 결과 종합 → 정리라는 여러 단계를 이어 밟습니다.

```
복잡한 일을 여러 라운드로 — "이번 주 가장 싼 서울→부산 항공권"

라운드1: [생각] 요일별 가격을 모은다  → [행동] 검색(월) → [관찰] 6만원
라운드2: [생각] 다른 요일도 확인     → [행동] 검색(수) → [관찰] 4.5만원
라운드3: [생각] 주말도 본다          → [행동] 검색(토) → [관찰] 8만원
라운드4: [생각] 수요일이 최저다 → 정리할 수 있다
   ▼
답: "수요일이 4.5만원으로 가장 쌉니다. 예약 링크: …"
```

이처럼 에이전트는 *한 번의 답*이 아니라 *여러 단계의 작업 흐름*을 굴립니다. 각 라운드의 관찰이 다음 라운드의 생각을 바꾸기 때문에, 미리 짠 고정 스크립트가 아니라 *상황에 적응하는* 수행이 가능합니다.

### 모델이 도구를 어떻게 배우나

모델이 도구를 *언제·어떻게* 호출할지는 어떻게 익힐까요? 한 갈래는 **Toolformer**처럼, 텍스트 안에 도구 호출을 끼워 넣는 법을 *자기지도 방식*으로 학습시키는 것입니다 [2] — 계산기·검색·번역기 호출을 스스로 삽입하도록 배운 뒤, 그게 정답 예측에 도움이 될 때만 남기는 식이죠. 다른 갈래는 정렬 단계(1번 글)에서 *도구 호출 형식*을 따르도록 다듬는 것입니다. 어느 쪽이든 핵심은, 모델이 "내가 직접 답할지, 도구를 부를지"를 *판단*하도록 만든다는 점입니다.

도구 사용이 왜 본질적으로 중요한지 한 번 더 짚겠습니다. LLM은 *언어의 통계*에는 강하지만 *정확한 연산*에는 약합니다. 큰 수의 곱셈을 "기억"으로 흉내 내다 틀리는 거죠. 반면 계산기는 곱셈을 *정확히* 합니다. 그래서 "모델이 직접 푼다"보다 "모델이 계산기를 *부른다*"가 더 정확합니다. 검색·코드 실행·데이터베이스 조회도 마찬가지입니다 — 모델은 *언제 어떤 도구가 필요한지 판단하는 지휘자* 역할을 하고, 정확성이 필요한 일은 전문 도구에 맡깁니다. 이 분업이 에이전트가 단순 챗봇보다 훨씬 믿을 만한 이유입니다.

## 방법론 — 멀티모달 (multimodal)

**멀티모달(multimodal)**은 텍스트 한 종류(모달리티)를 넘어 **이미지·음성·영상** 등 여러 형태를 함께 다루는 것입니다. 그런데 글과 그림은 형태가 완전히 다른데, 한 모델이 어떻게 둘을 함께 이해할까요?

핵심 아이디어는 4단계에서 이미 만났습니다 — **모든 것을 벡터로.**

- 텍스트는 임베딩으로 벡터가 됩니다.
- 이미지도 (전용 인코더, 보통 비전 트랜스포머를 거쳐) **같은 의미 공간의 벡터**로 바꿉니다.
- 그러면 "강아지 사진"의 벡터와 "강아지"라는 단어의 벡터가 *서로 가까이* 놓입니다.

```
   "강아지" (텍스트)   ──텍스트 인코더──▶ 벡터 ●
                                              ┐ 같은 공간에서
   [강아지 사진] (이미지) ──이미지 인코더──▶ 벡터 ●  서로 가까이 놓임
                                              ┘ → 글↔그림을 넘나들 수 있음
```

> 한 줄 원리: **서로 다른 형태(글·그림·소리)를 같은 벡터 공간에 모으면, 한 모델이 그것들을 함께 이해하고 넘나들 수 있다.**

이 "글과 이미지를 같은 공간에 정렬한다"는 발상을 대규모로 실현해 큰 영향을 준 것이 **CLIP**입니다 [3] — 인터넷에서 모은 4억 쌍의 (이미지, 설명) 데이터로, *맞는 이미지-설명 쌍은 가깝게, 틀린 쌍은 멀게* 학습시켜, 별도 라벨 학습 없이도 새 이미지를 분류(제로샷)할 수 있게 했습니다.

그 덕에 멀티모달 모델은 사진을 보고 설명하고, 그래프를 읽고, 손글씨를 옮기고, 음성을 알아듣습니다. 4단계 트랜스포머의 "모든 것을 벡터로 바꿔 어텐션으로 섞는다"는 발상이 *글 너머로* 확장된 것이죠. 입력만 여러 형태인 게 아니라, 출력도 글·그림·음성으로 낼 수 있으면 "멀티모달 입출력"이 됩니다.

여기서 트랜스포머의 범용성이 빛납니다. 4단계에서 봤듯, 트랜스포머는 "토큰(벡터)의 나열"이기만 하면 *무엇이든* 어텐션으로 섞을 수 있습니다. 그러니 이미지를 작은 패치(예: 16×16 픽셀)로 잘라 각 패치를 벡터(토큰)로 만들면, *이미지도 트랜스포머가 다룰 수 있는 토큰 나열*이 됩니다. 이 발상을 처음 본격화한 것이 비전 트랜스포머(ViT)입니다 [4] — 논문 제목 그대로 "이미지는 16×16 단어만큼의 가치가 있다"는 것이죠. 텍스트 토큰과 이미지 패치 토큰을 *한 줄에 섞어* 어텐션에 넣으면, 모델은 "사진의 이 부분이 질문의 이 단어와 관련 있다"를 어텐션으로 연결합니다. 같은 엔진(어텐션) 하나로 글과 그림을 함께 처리하는 것 — 이것이 멀티모달이 별도의 신기술이 아니라 *트랜스포머의 자연스러운 확장*인 이유입니다.

### 멀티모달이 여는 일들

이 확장이 실제로 무엇을 가능케 하는지 몇 가지만 봅시다.

- **이미지 설명·질의응답**: 사진을 보고 "이건 무엇이고 어디가 이상한가"를 답합니다. 의료 영상 보조, 사고 현장 분석 등에 쓰입니다.
- **문서·그래프 이해**: 표·차트·수식이 담긴 화면을 *그대로 읽어* 내용을 정리합니다. 텍스트로 옮기기 어려운 자료를 다룰 수 있습니다.
- **음성 대화**: 말을 알아듣고(음성→의미), 말로 답합니다(의미→음성). 손을 못 쓰는 상황이나 접근성 보조에 강력합니다.

핵심은 이 모든 게 *서로 다른 모델 여러 개*가 아니라, *같은 의미 공간을 공유하는 하나의 모델*로 점점 통합되고 있다는 흐름입니다. 글·그림·소리를 같은 공간에 모았기에, 한 모델이 "사진을 보고 글로 설명하고 음성으로 읽어 주는" 변환을 자연스럽게 넘나듭니다.

## 직관 — 비서의 진화

- **기본 LLM** = 박학다식하지만 *책상에 앉아 말만* 하는 상담가. 전화도 못 걸고, 인터넷도 못 뒤지고, 당신이 내민 사진도 못 봅니다.
- **에이전트** = *손발 달린 비서*. 검색하고, 계산하고, 예약을 넣고, 결과를 확인해 다음 일을 합니다. "생각만"이 아니라 "행동"이 가능해집니다.
- **멀티모달** = *눈과 귀까지 달린 비서*. 당신이 보여 준 사진과 들려준 음성, 보낸 그래프까지 이해합니다.

세 가지가 합쳐지면, *보고 듣고 도구를 써서 일을 수행하는* 조수에 가까워집니다. 그리고 여기에 앞 글들이 전부 모입니다 — 추론(2번 글)으로 계획을 세우고, RAG(4번 글)로 자료를 찾고, 도구로 행동하고, 멀티모달로 감각을 더하는 식으로요. 6단계의 기술들이 *따로 노는 조각*이 아니라 *하나의 시스템*으로 합쳐지는 지점이 바로 에이전트입니다.

```
하나의 시스템으로 합쳐지는 6단계 기술들

정렬(1) ── 사람이 원하는 방식으로 행동하도록
추론(2) ── 여러 단계 계획을 세움          ┐
RAG(4)  ── 외부 자료를 찾아 근거로 삼음    ├─▶ 에이전트
도구(5) ── 계산·검색·실행을 수행          │   (보고-찾고-생각하고-행동)
멀티모달(5) ── 사진·음성까지 이해          ┘
```

비유로 정리하면, 기본 LLM이 *책상에 묶인 상담가*였다면, 이 모든 것을 갖춘 에이전트는 *눈·귀·손발에 더해 자료실 출입증과 전화기까지 가진 비서*입니다. 능력의 차이는 모델을 키워서가 아니라, 모델을 *어떻게 둘러싸느냐*에서 나옵니다 — 이것이 6단계 후반부의 핵심 메시지입니다.

## 흔한 오해

- **"에이전트는 새로운 종류의 모델이다"가 아닙니다.** 보통은 같은 LLM에 *도구를 붙이고 [생각-행동-관찰] 루프로 감싼* 시스템입니다 [1]. 모델 자체보다 *둘러싼 구조*가 핵심이라, 같은 모델도 어떻게 감싸느냐에 따라 에이전트가 되기도 안 되기도 합니다.
- **"도구를 많이 줄수록 똑똑해진다"가 아닙니다.** 도구가 많아지면 *언제 무엇을 쓸지 고르는* 판단이 어려워지고, 잘못된 도구를 부르거나 엉뚱한 인자를 넘기는 실수도 늘어 오히려 불안정해질 수 있습니다.
- **"멀티모달은 곧 이미지 생성을 뜻한다"가 정확하지 않습니다.** 멀티모달은 여러 형태를 *이해(입력)*하거나 *생성(출력)*하는 것 전반을 가리킵니다. 사진을 '읽고 설명하는' 것도 어엿한 멀티모달입니다.
- **"에이전트는 알아서 다 잘한다"가 아닙니다.** 루프가 길어질수록 중간에서 *한 번 어긋난 판단*이 누적돼 엉뚱한 결과로 흐를 수 있습니다. 그래서 실무 에이전트는 단계 수를 제한하고, 위험한 행동(결제·삭제 등)에는 사람의 확인을 끼워 넣습니다. 이 신뢰성 문제는 6번 글의 한계와 직결됩니다.

## 한 줄 요약

> **에이전트는 LLM에 손발(도구)을 달아 [생각→행동→관찰] 루프로 일을 수행하게 하고, 멀티모달은 눈·귀(이미지·음성)를 달아 준다.**
> 둘 다 "모든 것을 같은 벡터 공간에 모은다"는 트랜스포머의 발상 위에서, 글만 다루던 상자를 행동과 감각으로 넓힌다 — 다만 루프가 길고 도구가 많아질수록 신뢰성 관리가 관건이 된다.
