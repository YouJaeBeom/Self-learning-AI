# 1. 정렬 — RLHF와 DPO

5단계에서 LLM은 인터넷의 방대한 글로 "다음 단어"를 귀신같이 맞히게 되었습니다. 그런데 여기서 불편한 진실 하나와 마주칩니다. **"다음 단어를 잘 맞히는 것"과 "사람에게 도움이 되고 안전한 답을 하는 것"은 전혀 다른 문제입니다.** 인터넷에는 친절한 글만 있는 게 아니라 무례한 글, 틀린 글, 위험한 글도 가득하니까요. 그 모든 걸 똑같이 잘 흉내 내도록 학습된 모델을, *사람이 쓰고 싶은 조수*로 바꾸는 마지막 단계 — 그것이 **정렬(alignment)**입니다.

이 글은 6단계의 출발점입니다. 정렬을 이해하면, 왜 ChatGPT나 Claude 같은 모델이 그냥 거대한 자동완성기가 아니라 *대화 상대*처럼 느껴지는지가 보입니다.

---

## 목적 — 똑똑함을 '쓸모 있음'으로 바꾸기

사전학습만 끝낸 모델을 **베이스 모델(base model)**이라고 부릅니다. 이 모델은 글을 *이어 쓰는* 데 특화돼 있을 뿐, 질문에 *답하도록* 만들어지지 않았습니다. 그래서 베이스 모델에게 "프랑스의 수도는?"이라고 물으면, 답을 주는 대신 "프랑스의 수도는? 독일의 수도는? 이탈리아의 수도는?" 하고 *비슷한 질문 목록을 이어 쓰는* 황당한 일이 자주 벌어집니다. 인터넷에는 그런 형식의 글(문제집, 퀴즈 목록)도 많기 때문에, 모델 입장에서는 그게 가장 자연스러운 "다음 단어"일 수 있는 거죠.

> 정렬의 목적: 모델을 **"사람이 원하는 답·도움 되는 답·안전한 답"**에 맞추는 것. 능력(무엇을 할 수 있는가)을 늘리는 게 아니라, *태도와 방향*(무엇을 해야 하는가)을 잡아 주는 일입니다.

이것이 5단계에서 예고한 "사전학습 → 파인튜닝 → **정렬**" 3박자의 마지막 박자입니다. 첫 박자(사전학습)에서 *능력*이 만들어지고, 마지막 박자(정렬)에서 *태도*가 만들어진다고 보면 깔끔합니다. 정렬에서 흔히 드는 목표 세 가지를 영어 머리글자로 **3H** — Helpful(도움 됨), Honest(정직함), Harmless(해롭지 않음) — 라고 부릅니다 [4].

---

## 핵심 개념 — 왜 어려운가

"좋은 답"은 **규칙으로 적기가 거의 불가능합니다.** 0단계의 고양이 사진 문제와 똑같은 구조입니다. "고양이를 픽셀 규칙으로 정의하라"가 불가능했듯, "공손하고 정확하고 도움 되게 답하라"를 if-else 코드로 쓸 방법이 없습니다. 무엇이 "공손한가"는 상황과 맥락에 따라 무한히 갈라지니까요.

그래서 2단계에서 배운 머신러닝의 근본 발상을 다시 꺼냅니다 — **규칙을 못 적으면, 예시를 모아 배우게 하자.** 다만 이번에 모으는 예시는 "정답"이 아니라 **사람의 선호(preference)**입니다.

> 핵심 장치: 같은 질문에 대한 여러 답을 사람에게 보여 주고 **"어느 쪽이 더 나은가"**(A가 좋은가, B가 좋은가)를 고르게 합니다. 이 *선호 데이터(preference data)*가 정렬의 연료입니다.

왜 "절대 점수"가 아니라 "둘 중 비교"일까요? 사람에게 "이 답은 100점 만점에 몇 점?"이라고 물으면 채점자마다 기준이 제각각이라 들쭉날쭉합니다. 하지만 "A와 B 중 어느 쪽이 나아?"라고 물으면 훨씬 일관되게 답합니다. 비교는 절대 평가보다 *쉽고 신뢰할 만한* 신호라서, 정렬 데이터는 대부분 이 쌍대 비교(pairwise comparison) 형태로 모읍니다.

---

## 방법론 — RLHF: 사람 피드백 기반 강화학습

**RLHF(Reinforcement Learning from Human Feedback)**는 1단계에서 배운 강화학습(보상으로 배우기)을 LLM에 적용한 것입니다. 이 방법을 대규모 언어 모델에 처음으로 제대로 적용해 대중화한 작업이 OpenAI의 InstructGPT입니다 [1]. 세 단계로 진행됩니다.

```
① 지도 파인튜닝(SFT)
   사람이 쓴 모범 답안(질문→좋은 답)으로 베이스 모델을 파인튜닝.
   "질문에는 답을 한다"는 기본 대화법을 먼저 익히게 함.
        │
        ▼
② 보상 모델(RM) 학습
   같은 질문에 SFT 모델이 답을 여러 개 생성 → 사람이 순위를 매김(A>B>C)
   이 선호 데이터로, "답에 점수를 매기는" 별도의 보상 모델을 학습.
        │
        ▼
③ 강화학습(PPO)
   LLM(정책)이 답을 생성 → 보상 모델이 점수를 매김 →
   점수가 높아지는 방향으로 LLM을 업데이트. (이 ③을 수없이 반복)
```

### 보상 모델 — 사람을 대신하는 자동 채점기

**보상 모델(reward model, RM)**은 "사람이라면 이 답에 몇 점을 줄까?"를 *대신 예측해 주는* 모델입니다. 강화학습은 매 단계마다 수많은 답에 점수를 매겨야 하는데, 사람이 그걸 일일이 채점할 수는 없습니다. 그래서 ②에서 모은 사람의 선호로 *사람의 취향을 학습한 자동 채점기*를 만들어 두고, ③에서는 이 채점기를 사람 대신 씁니다.

보상 모델을 학습하는 손실함수의 핵심 직관은 이렇습니다 — "사람이 A를 B보다 선호했다면, RM이 매기는 점수도 A가 B보다 높아지도록" 만듭니다. 수식으로는 두 점수의 차이를 시그모이드(2단계)에 넣어, 선호된 답의 점수가 높을수록 손실이 작아지게 합니다 (Bradley-Terry 모델 기반).

### 강화학습 단계 — PPO와 KL 고삐

③에서 LLM을 업데이트하는 알고리즘으로는 보통 **PPO(Proximal Policy Optimization)**를 씁니다 [3]. PPO는 "한 번에 너무 크게 정책을 바꾸지 않도록 *완만하게* 업데이트하는" 강화학습 기법으로, 학습이 폭주하지 않게 잡아 주는 안전벨트 역할을 합니다.

그런데 보상만 무작정 높이려 들면 한 가지 함정에 빠집니다. 모델이 *보상 모델의 허점*을 파고들어, 사람이 보기엔 이상하지만 RM 점수만 높은 답(예: 특정 단어를 무의미하게 반복)으로 망가지는 거죠. 이걸 막으려고 목적함수에 **KL 페널티(KL penalty)**라는 고삐를 답니다.

> RLHF 목적함수의 직관: **"보상(사람이 좋아할 점수)은 높이되, 원래 SFT 모델에서 너무 멀어지지는 마라."** 뒷부분(KL 페널티)이 없으면 점수만 노리다 이상한 말투로 무너지기 때문에, 원본과의 거리(KL 발산)에 비례하는 벌점을 둬서 *적당한 선에서* 멈추게 합니다.

InstructGPT 논문의 인상적인 결과 하나 — 파라미터가 100배 이상 작은 1.3B RLHF 모델의 답을, 사람들이 175B 베이스 GPT-3의 답보다 더 선호했습니다 [1]. 정렬이 단순한 미세 조정이 아니라 *체감 품질을 통째로 바꾸는* 단계임을 보여 준 셈입니다.

---

## 수학 — DPO: 보상 모델 없이 직접 최적화

RLHF는 강력하지만 **복잡하고 불안정**합니다. 보상 모델을 따로 학습하고(②), 까다로운 강화학습 루프를 돌려야 하며(③), 그 과정에서 모델 여러 개(정책·보상·참조)를 동시에 메모리에 올려야 하니까요. 학습이 삐끗하는 지점도 많습니다.

**DPO(Direct Preference Optimization, 직접 선호 최적화)**는 이걸 통째로 단순화한 방법입니다 [2]. 논문 제목이 핵심을 찌릅니다 — *"당신의 언어 모델은 사실 비밀리에 보상 모델이다."*

> 핵심 아이디어: **별도의 보상 모델과 강화학습을 건너뛰고, 선호 데이터로 LLM을 *직접* 학습한다.**

### 무엇이 일어나는가

발상은 이렇습니다. "사람이 A를 B보다 선호했다"는 정보를, **A의 확률(가능도)은 올리고 B의 확률은 내리는** 단 하나의 손실함수로 바꿔 버립니다. 강화학습 루프 대신, 2단계에서 배운 *지도학습*처럼 손실을 한 번 정의하고 경사하강법으로 내려가면 끝입니다.

핵심은 수학적 증명입니다. Rafailov et al.(2023)은 RLHF가 *결국 최적화하려던 목표*(보상 최대화 + KL 페널티)를, 보상 모델이라는 중간 단계 없이 *직접 푸는* 등가의 손실함수로 다시 쓸 수 있음을 보였습니다. 보상 모델을 명시적으로 만드는 대신, **언어 모델 자신이 암묵적인 보상 모델 역할**을 하도록 식을 재배열한 것이죠. 그래서 RLHF의 두세 단계가 *한 단계의 분류 문제*로 압축됩니다.

> 한 줄 풀이: "선호된 답의 가능도는 높이고, 거부된 답의 가능도는 낮추되, 참조 모델에서 과하게 벗어나지 않게" — RLHF의 ②③을 *한 번의 지도학습*으로 접어 넣은 셈입니다. KL 고삐의 정신은 DPO 손실 안에도 그대로 살아 있습니다.

### RLHF vs DPO 비교

| 항목 | RLHF | DPO |
|---|---|---|
| 보상 모델 | **필요** (②에서 따로 학습) | **불필요** (모델이 암묵적 보상 모델) |
| 학습 방식 | 강화학습 (PPO, 루프 반복) | 지도학습에 가까움 (손실 한 번 정의) |
| 안정성 | 상대적으로 불안정·민감 | 단순·안정 |
| 필요한 모델 수 | 정책·보상·참조 등 여럿 | 정책·참조 둘 |
| 세밀한 제어 | 유리 (보상 설계로 조절 가능) | 상대적으로 제한적 |
| 한 줄 | 채점기를 만들고 점수를 올린다 | 선호를 손실로 바꿔 바로 배운다 |

DPO는 등장 이후 *간단하고 재현하기 쉽다*는 이유로 오픈소스 모델 정렬의 표준 도구 중 하나가 됐습니다. 다만 "DPO가 RLHF를 완전히 대체했다"고 보기는 어렵고, 둘은 같은 목표를 다른 길로 가는 *대안 관계*에 가깝습니다.

## 직관 — 요리사 길들이기

갓 사전학습된 모델은 *재료는 다 아는데 손님 입맛은 모르는* 요리사입니다. 칼질도 잘하고 레시피도 외우고 있지만, 정작 "이 식당 손님들이 뭘 좋아하는지"는 모릅니다.

- **RLHF**: 먼저 *맛 평론가(보상 모델)*를 한 명 따로 키웁니다. 손님들의 평가를 잔뜩 보여 줘서 "이 식당 입맛"을 학습시키는 거죠. 그다음 요리사가 요리를 내면 평론가가 점수를 주고, 요리사는 그 점수를 높이려 계속 고칩니다. 단계가 많고, 평론가가 삐끗하면(보상 해킹) 요리사가 *평론가만 속이는* 이상한 요리로 망가질 수 있습니다. 그래서 "원래 레시피에서 너무 벗어나지 마"라는 고삐(KL)를 함께 채웁니다.
- **DPO**: 평론가를 따로 키우는 과정을 통째로 생략합니다. 대신 **"손님이 A 접시를 B보다 좋아했다"는 기록**을 요리사에게 직접 주며 "이런 건 더, 저런 건 덜"이라고 바로 가르칩니다. 평론가라는 중간 다리가 없으니 더 간단하고, 흔들릴 구석도 적습니다.

같은 목적지(손님이 만족하는 요리)에, RLHF는 평론가를 거쳐 돌아가고 DPO는 손님 기록으로 곧장 갑니다.

---

## 흔한 오해

- **"정렬은 모델을 더 똑똑하게 만든다"가 아닙니다.** 정렬은 지식이나 능력을 늘리는 게 아니라, 이미 가진 능력을 *사람이 원하는 방향으로 끌어내고 다듬는* 것입니다. 오히려 너무 세게 정렬하면 일부 능력이 약간 깎이기도 하는데, 이를 **정렬 세금(alignment tax)**이라 부릅니다 (Ouyang et al., 2022는 학습 데이터를 섞어 이 손실을 줄이는 PPO-ptx를 제안했습니다).
- **"DPO가 RLHF보다 항상 낫다"가 아닙니다.** DPO는 더 단순하고 안정적이지만, RLHF가 보상 설계를 통해 더 유연하고 세밀한 제어에 유리한 경우도 많습니다. 둘은 *같은 목표를 다른 길로* 가며, 데이터·예산·목표에 따라 골라 씁니다.
- **"보상 모델은 정답을 안다"가 아닙니다.** 보상 모델은 *사람의 평균적 취향을 흉내 낸* 근사일 뿐, 완벽하지 않습니다. 이 허점을 모델이 악용해 점수만 높이는 헛수를 두는 문제를 **보상 해킹(reward hacking)**이라 합니다. 6번 글의 '정렬 문제'와 곧장 이어지는 한계입니다.
- **"RLHF의 선호 데이터는 정답 라벨이다"가 아닙니다.** 선호 데이터는 "둘 중 어느 쪽이 나은가"라는 *상대적 비교*일 뿐, "이게 유일한 정답"이라는 절대 라벨이 아닙니다. 그래서 사람마다 선호가 갈리는 영역에서는 정렬도 흔들릴 수밖에 없습니다.

---

## 한 줄 요약

> **정렬은 똑똑한 모델을 '사람이 원하고 안전한 답'에 맞추는 단계다.**
> RLHF는 사람의 선호로 보상 모델을 만들어 PPO 강화학습으로 점수를 올리되 KL 고삐로 폭주를 막고, DPO는 보상 모델 없이 선호 데이터로 직접 최적화한다 — 더 단순하고 안정적이지만, 둘은 같은 목표의 다른 길이다.
