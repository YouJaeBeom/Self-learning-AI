# 4. RAG — 검색으로 환각 줄이기

LLM에는 고질병이 하나 있습니다. **모르는 것도 아는 척, 그럴듯하게 지어냅니다.** 존재하지 않는 논문을 진짜처럼 인용하고, 틀린 날짜를 자신만만하게 말하죠. 이걸 **환각(hallucination)**이라 부릅니다. 그리고 **RAG(검색 증강 생성, Retrieval-Augmented Generation)**는 이 문제를 정면으로 공략하는, 오늘날 가장 널리 쓰이는 기법입니다.

발상은 의외로 단순합니다 — *답하기 전에 자료를 찾아보게 하자.* 이 단순한 발상이 어떻게 환각을 다스리는지, 그 안에 어떤 수학이 숨어 있는지 봅니다.

---

## 목적 — '기억'만으로 답하는 위험

기본 LLM은 학습 때 본 내용을 **가중치 속 기억**으로만 답합니다. 학습이 끝나면 그 지식이 *모델 안에 통째로 박제*되는 셈이죠. 여기엔 두 가지 근본 약점이 있습니다.

1. **최신성**: 학습이 끝난 시점 이후의 일은 *전혀* 모릅니다. 지식이 특정 시점에 얼어붙어 있어, "어제 일어난 일"은 답할 수 없습니다.
2. **정확성**: 세부 사실(숫자·이름·인용·날짜)을 *흐릿하게 기억*해 그럴듯하게 메꿉니다. 모델은 "정확한 사실"이 아니라 "그럴듯한 다음 단어"를 좇기 때문에, 빈틈을 만나면 지어내는 쪽으로 기울죠. 이것이 환각의 뿌리입니다.

> RAG의 목적: 모델이 **외부의 믿을 수 있는 자료를 찾아 근거로 삼아** 답하게 만드는 것. 지식을 *모델 안*이 아니라 *모델 바깥의 검색 가능한 창고*에 두는 것이 핵심입니다.

비유하자면 *오픈북 시험*입니다. 머릿속 흐릿한 기억에만 의존하지 않고 *책을 펴서 확인하고* 답하니, 지어낼 이유 자체가 줄어듭니다. RAG라는 개념과 이름을 처음 제시한 것이 Lewis et al.(2020)입니다 — 파라미터에 저장된 지식(parametric memory)과 외부 문서(non-parametric memory)를 결합하면 더 사실적인 답을 낸다는 걸 보였습니다.

---

## 핵심 개념 — RAG의 두 박자

RAG는 이름 그대로 **검색(Retrieval) + 생성(Generation)**의 두 박자입니다.

```
질문: "우리 회사 환불 규정은?"
   │
   ▼
① 검색(Retrieval)
   질문과 의미가 가까운 문서 조각을 자료 창고(벡터 DB)에서 찾아옴
   → "환불은 구매 후 14일 이내, 미개봉 시 전액 환불됩니다…"
   → "교환은 영수증 지참 시 30일 이내 가능…"  (관련 조각 여러 개)
   │
   ▼
② 생성(Generation)
   찾아온 문서들을 질문과 함께 프롬프트에 넣어, 그것을 근거로 LLM이 답 생성
   "아래 자료를 참고해 답하라: [찾아온 문서] / 질문: 환불 규정은?"
   │
   ▼
답: "구매 후 14일 이내 미개봉 상품은 전액 환불됩니다." (찾아온 출처에 근거)
```

핵심은 ②에서 LLM이 *자기 흐릿한 기억*이 아니라 **방금 찾아온 문서**를 보고 답한다는 점입니다. 그래서 근거가 분명하고, "이 답은 어느 문서의 몇 번째 문단에서 나왔다"는 출처를 댈 수도 있습니다. 환각이 일어날 틈이 그만큼 좁아집니다.

이 구조의 또 다른 장점 — **지식을 바꾸려고 모델을 다시 학습할 필요가 없습니다.** 회사 규정이 바뀌면 *창고의 문서만 갈아 끼우면* 됩니다. 수백억 가중치를 재학습하는 것과 비교하면 비교할 수 없이 싸고 빠릅니다.

### 문서를 '조각'으로 나누는 이유

위에서 "문서 조각"이라는 말을 썼습니다. 왜 문서 전체가 아니라 조각일까요? 두 가지 이유입니다. 첫째, LLM의 프롬프트 길이에는 한계가 있어 *책 한 권을 통째로* 넣을 수 없습니다. 둘째, 질문과 관련된 건 보통 문서의 *한두 문단*뿐인데, 무관한 부분까지 다 넣으면 잡음만 늘어납니다. 그래서 문서를 적당한 크기(예: 몇 문단 단위)로 잘라 **청크(chunk)**로 만들어 두고, 질문에 *정말 관련된 청크만* 골라 넣습니다. 청크를 너무 크게 자르면 잡음이 섞이고, 너무 작게 자르면 맥락이 끊깁니다 — 이 *조각 크기* 조절이 RAG 품질의 숨은 변수입니다.

## 수학 — 의미로 검색하기 (코사인 유사도)

"의미가 가까운 문서를 찾는다"를 컴퓨터가 어떻게 *숫자로* 할까요? 여기서 4단계의 **임베딩**이 다시 주역으로 등장합니다.

### 벡터로 바꿔 저장하고, 벡터로 검색한다

```
[준비 단계 — 미리 한 번]
모든 문서 조각 ──임베딩──▶ 벡터 ──저장──▶ 벡터 데이터베이스
  "환불은 14일…"            [0.2, -0.8, …]      (수백만 개 벡터 보관)

[질의 단계 — 질문이 올 때마다]
질문 "환불 규정?" ──임베딩──▶ 질문 벡터 [0.3, -0.7, …]
                                    │
                                    ▼  가장 가까운 벡터 top-k 찾기
                          ["환불은 14일…", "교환은 30일…"]  → ②로 전달
```

- 모든 문서 조각을 미리 **임베딩 벡터**로 바꿔 **벡터 데이터베이스(vector DB)**에 저장해 둡니다.
- 질문이 오면 질문도 *같은 임베딩 방식*으로 벡터로 바꿉니다.
- 질문 벡터와 가장 **가까운(유사한)** 문서 벡터 몇 개(top-k)를 찾습니다.

### 가까움을 재는 자 — 코사인 유사도

이 "가까움"을 재는 대표 척도가 **코사인 유사도(cosine similarity)**입니다.

`cos_sim(a, b) = (a · b) / (|a| · |b|)`

> 이 식이 재는 것: 두 벡터가 이루는 **각도**입니다. 같은 방향이면 1(매우 비슷), 직각이면 0(무관), 반대 방향이면 −1(반대). 벡터의 *크기(길이)는 무시*하고 *방향=의미*만 봅니다. 4단계의 "내적 = 유사도"를 길이로 정규화해 −1~1 척도로 다듬은 것입니다.

왜 크기를 무시할까요? 긴 문서든 짧은 문서든 *주제(방향)가 같으면* 비슷하다고 봐야 하기 때문입니다. 코사인 유사도는 "얼마나 큰가"가 아니라 "어느 쪽을 향하는가"만 보므로, 문서 길이에 휘둘리지 않고 의미를 잽니다.

이 덕에 **단어가 하나도 겹치지 않아도 의미가 통하면** 찾아냅니다. "환불"로 검색해도 "반품·교환 정책" 문서를 찾아오는 식이죠. 이것이 단순 키워드 일치를 넘어선 **의미 검색(semantic search)**입니다. 이런 밀집 임베딩 기반 검색이 전통적 키워드 검색(BM25)보다 우수함을 보인 대표 작업이 Karpukhin et al.(2020)의 DPR(Dense Passage Retrieval)입니다. 실무에서는 키워드 검색과 의미 검색을 *함께 쓰는* 하이브리드 방식도 흔합니다.

키워드 검색과 의미 검색의 차이를 한 표로 정리하면 이렇습니다.

| | 키워드 검색 (BM25 등) | 의미 검색 (임베딩) |
|---|---|---|
| 무엇을 비교 | 단어 일치 (글자) | 의미 방향 (벡터) |
| 강점 | 정확한 용어·고유명사·코드 | 동의어·다른 표현·맥락 |
| 약점 | "환불"≠"반품"을 못 잇음 | 정확한 숫자·이름 매칭이 약할 수 있음 |
| 비유 | 색인 카드 뒤지기 | 뜻이 비슷한 책 찾기 |

그래서 둘을 합친 하이브리드 검색이 강력합니다 — *정확한 단어*는 키워드로 잡고, *비슷한 뜻*은 임베딩으로 잡는 식으로 서로의 약점을 메웁니다.

## 직관 — 오픈북 시험

- **기본 LLM** = *암기 시험*을 보는 학생. 머릿속 기억만으로 답합니다. 확실히 외운 건 잘 맞히지만, 가물가물한 세부는 *그럴듯하게 지어냅니다*(환각).
- **RAG** = *오픈북 시험*을 보는 학생. 질문을 받으면 *관련 페이지를 펼쳐* 확인하고 답합니다. 지어낼 필요가 줄고, 교재가 최신판이면 최신 내용도, 회사 내부 자료라면 전용 지식도 반영합니다.

핵심 통찰은 이것입니다 — **RAG는 모델을 *더 똑똑하게* 만드는 게 아니라, 답의 근거를 모델 바깥의 믿을 만한 자료로 옮기는 것입니다.** 그래서 모델을 다시 학습시키지 않고도 새 지식이나 전용 지식(사내 문서, 최신 뉴스)을 붙일 수 있고, "이 답의 출처는 여기"라고 댈 수 있습니다.

이 발상이 실무에서 강력한 이유가 하나 더 있습니다 — **모델은 그대로 두고, 지식만 분리해 관리**할 수 있다는 점입니다. 같은 LLM 하나로 법률 회사에는 판례 창고를, 병원에는 진료 지침 창고를, 쇼핑몰에는 상품·정책 창고를 붙이면, *각 분야에 특화된 조수*가 됩니다. 모델을 분야마다 따로 학습시킬 필요가 없으니, 비용과 시간이 크게 줄죠. "능력은 공통, 지식은 갈아끼움"이라는 분업이 RAG가 기업에서 가장 먼저 자리 잡은 이유입니다.

## 방법론 — 파인튜닝과의 비교, 그리고 한계

지식을 모델에 넣는 방법은 크게 두 가지입니다. 어느 쪽을 쓸지 헷갈릴 때가 많아 정리해 둡니다.

| 항목 | RAG (검색으로 붙이기) | 파인튜닝 (가중치에 새기기) |
|---|---|---|
| 지식이 사는 곳 | 모델 *바깥* 창고 (벡터 DB) | 모델 *안* 가중치 |
| 지식 갱신 | 문서만 갈아 끼우면 됨 (즉시) | 재학습 필요 (비싸고 느림) |
| 출처 표시 | 가능 ("이 문서 근거") | 어려움 (가중치에 녹아듦) |
| 잘하는 것 | 사실·최신성·전용 자료 | 말투·형식·도메인 *감각* |
| 비용 | 검색 인프라 필요, 학습은 불필요 | 학습 비용 발생 |

대략적인 지침 — **"무엇을 아는가(사실)"는 RAG로, "어떻게 말하는가(스타일·형식)"는 파인튜닝으로.** 둘은 배타적이지 않고 함께 쓰기도 합니다.

실제 RAG 파이프라인은 위 두 박자보다 조금 더 깁니다. 전체를 한눈에 그리면 이렇습니다.

```
[준비] 문서 → 청크 분할 → 임베딩 → 벡터 DB에 저장
                                       │
[질의] 질문 → 임베딩 → 유사도 검색(top-k) → 관련 청크
                                       │
       (선택) 재정렬(rerank): 찾아온 청크를 한 번 더 정밀 점수로 추림
                                       │
       프롬프트 조립: [질문 + 관련 청크] → LLM → 근거 있는 답 + 출처
```

여기서 **재정렬(rerank)** 단계가 종종 추가됩니다. 1차 유사도 검색은 빠르지만 거칠어서, 후보를 넉넉히(예: 20개) 뽑은 뒤 *더 정밀한 모델*로 다시 점수를 매겨 *진짜 관련된 소수*(예: 3개)만 남기는 것이죠. "빠르게 많이 거른 뒤, 정밀하게 추린다"는 2단계 전략이 검색 품질을 끌어올립니다.

RAG가 만능은 아닙니다. 파이프라인이 *검색*과 *생성* 두 단계로 나뉘어 있으니, 둘 중 어디서든 어긋날 수 있습니다.

- **검색이 빗나가면**: 관련 없는 문서를 찾아오면, LLM은 *엉뚱한 근거*로 답하거나 그 문서를 무시하고 다시 지어냅니다. "쓰레기를 넣으면 쓰레기가 나온다"는 원칙이 그대로 적용됩니다.
- **생성이 무시하면**: 좋은 문서를 줘도 모델이 *제 기억을 고집*해 지어내는 경우가 있습니다. 특히 찾아온 자료가 모델의 기존 믿음과 충돌할 때 그렇습니다.
- **긴 문맥의 함정**: 찾아온 자료가 많아 프롬프트가 길어지면, 3번 글의 "중간에서 길을 잃는다"[3]가 발동해 핵심 자료를 놓칠 수 있습니다. 그래서 *적게, 정확하게* 찾아오는 검색 품질이 중요합니다.

---

## 흔한 오해

- **"RAG는 환각을 완전히 없앤다"가 아닙니다.** *줄일* 뿐입니다. 잘못된 문서를 찾아오거나, 좋은 문서를 줘도 모델이 무시하고 지어내면 여전히 틀립니다. RAG는 환각을 0으로 만드는 마법이 아니라, *근거를 외부로 옮겨 위험을 낮추는* 장치입니다.
- **"RAG는 파인튜닝과 같다"가 아닙니다.** 파인튜닝은 지식을 *가중치에 새겨 넣고*, RAG는 지식을 *바깥 창고에 두고 그때그때 찾아옵니다.* 갱신·출처 표시는 RAG가, 말투·형식 학습은 파인튜닝이 유리합니다.
- **"검색은 단어가 같아야 된다"가 아닙니다.** 임베딩 기반 의미 검색은 *표현이 달라도 뜻이 통하면* 찾습니다[2]. 이게 키워드 검색을 넘어 코사인 유사도를 쓰는 이유입니다.
- **"문서를 많이 넣을수록 좋다"가 아닙니다.** 관련 없는 문서가 섞이거나 프롬프트가 너무 길어지면 오히려 답이 나빠집니다. 검색 품질(정확히 관련된 소수를 찾기)이 양보다 중요합니다.

---

## 한 줄 요약

> **RAG는 "답하기 전에 외부 자료를 검색해 근거로 삼는" 오픈북 방식이다.**
> 문서와 질문을 임베딩해 코사인 유사도로 의미가 가까운 자료를 찾아오고(검색), 그것을 프롬프트에 넣어 근거 삼아 답하게 해(생성) 환각을 줄이며 최신·전용 지식을 모델 재학습 없이 붙인다 — 다만 검색이 빗나가면 효과도 무너진다.
