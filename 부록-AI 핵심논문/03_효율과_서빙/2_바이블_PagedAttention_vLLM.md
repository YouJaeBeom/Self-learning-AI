# KV 캐시를 페이지처럼 — PagedAttention / vLLM

> **Efficient Memory Management for Large Language Model Serving with PagedAttention (vLLM)** — Kwon et al., 2023, SOSP (arXiv:2309.06180)
> 원문: https://arxiv.org/abs/2309.06180 · 분류: 바이블

LLM을 학습하는 것과 수많은 사용자에게 동시에 서빙하는 것은 전혀 다른 공학 문제입니다. 학습은 처리량(throughput)이 핵심이지만, 서빙은 거기에 더해 지연시간과 메모리 효율, 그리고 동시 요청 수를 함께 잡아야 합니다. 이 서빙의 숨은 병목이 바로 KV 캐시 메모리입니다. 이 논문은 운영체제(OS)의 가상 메모리 페이징이라는 수십 년 된 아이디어를 LLM 서빙에 빌려 와, KV 캐시의 메모리 낭비를 거의 없앴습니다. 그렇게 탄생한 vLLM은 오늘날 가장 널리 쓰이는 오픈소스 서빙 엔진 중 하나가 되었습니다.

## 배경: KV 캐시와 메모리 단편화

자기회귀 생성에서 모델은 토큰을 하나씩 만들고, 매번 이전 모든 토큰의 Key·Value 벡터를 다시 참조합니다. 이를 매번 재계산하면 낭비이므로, 한 번 계산한 Key·Value를 저장해 두는데 이것이 KV 캐시입니다. 문제는 이 캐시가 생성이 진행될수록 계속 커진다는 점입니다. 요청 하나의 KV 캐시는 수 GB에 이를 수 있습니다.

기존 서빙 시스템은 이 KV 캐시를 위해 **연속된 큰 메모리 덩어리**를 미리 잡았습니다. 그런데 요청마다 최종 길이가 다릅니다. 어떤 요청은 2,048 토큰까지 가능성을 보고 큰 공간을 잡아 뒀는데 실제로는 100토큰에서 끝나면, 나머지 공간이 통째로 낭비됩니다. 이것이 내부 단편화(internal fragmentation)입니다. 여기에 더해 서로 다른 크기의 덩어리들이 들어찼다 빠지면서 생기는 외부 단편화도 발생합니다. 논문은 기존 시스템이 KV 캐시 메모리의 60~80%를 낭비하고 있었다고 보고합니다. 메모리가 낭비되면 동시에 처리할 수 있는 요청 수가 줄고, 그만큼 처리량이 떨어집니다.

> 핵심 질문: KV 캐시 메모리를 유연하게 관리해 낭비를 없애고, 같은 GPU로 더 많은 요청을 동시에 처리할 수 없을까?

## 핵심 아이디어: KV 캐시를 페이지처럼 쪼갠다

PagedAttention의 핵심은 KV 캐시를 OS의 가상 메모리 페이지처럼 작은 고정 크기 블록으로 쪼개 관리하는 것입니다.

> 비유 — 사물함 대 통창고. 짐(KV 캐시)을 사람마다 큰 통창고 한 칸씩 통째로 빌려주면, 짐이 적은 사람의 칸은 텅 비어 낭비됩니다. 대신 작은 사물함 여러 개로 쪼개 필요한 만큼만 배정하면, 빈 공간이 거의 없습니다. 사물함이 흩어져 있어도 어느 사물함이 누구 것인지 적은 목록(페이지 테이블)만 있으면 됩니다.

## 어떻게 작동하나

OS가 프로세스의 가상 주소를 물리 주소로 매핑하는 페이지 테이블을 두듯이, vLLM은 각 요청의 논리적 KV 블록을 물리적 블록에 매핑하는 블록 테이블을 둡니다.

```
요청의 토큰들 → 논리 블록 0,1,2,…  (연속)
블록 테이블   → 물리 블록 7, 3, 19,…  (메모리 어디에 흩어져 있어도 됨)
```

이 구조가 주는 이점은 두 가지입니다. 첫째, 블록을 필요할 때마다 하나씩 할당하므로 미리 큰 공간을 잡을 필요가 없어 내부 단편화가 거의 사라집니다. 논문은 낭비를 4% 미만으로 줄였다고 보고합니다. 둘째, 같은 내용의 블록을 여러 요청이 공유할 수 있습니다. 예를 들어 여러 요청이 같은 시스템 프롬프트로 시작하면, 그 공통 부분의 KV 블록을 한 벌만 두고 공유하다가(copy-on-write) 내용이 달라지는 시점에 복사합니다. 병렬 샘플링이나 빔 서치처럼 한 프롬프트에서 여러 출력을 만들 때 특히 큰 절약이 됩니다.

```
어텐션 결과:  바뀌지 않음 (정확히 동일)
메모리 낭비:  60~80% → 4% 미만
처리량:       기존 대비 2~4배
```

이 메모리 절약은 더 많은 요청을 한 배치에 담을 수 있게 하고, 연속 배칭(continuous batching)과 결합해 GPU를 쉴 틈 없이 돌립니다. 중요한 것은 어텐션의 계산 결과 자체는 전혀 바뀌지 않는다는 점입니다. 바뀐 것은 메모리 관리 방식뿐입니다.

| 항목 | 기존 서빙 | PagedAttention(vLLM) |
|---|---|---|
| KV 캐시 할당 | 연속 큰 덩어리 미리 잡음 | 작은 블록 필요 시 할당 |
| 메모리 낭비 | 60~80% | 4% 미만 |
| 블록 공유 | 어려움 | 공통 접두사 공유 가능 |
| 처리량 | 기준 | 2~4배 |

## 임팩트: 왜 바이블이 되었나

- 이 기법으로 만든 **vLLM**은 가장 널리 쓰이는 오픈소스 LLM 서빙 엔진 중 하나가 되었습니다. NVIDIA·AMD GPU는 물론 여러 클라우드에서 지원되며, 사실상 표준 서빙 스택의 하나로 자리 잡았습니다.
- LLM 서빙을 모델 품질이 아니라 **시스템 공학의 문제**로 재정의했습니다. "어떻게 더 잘 학습하나"만큼이나 "어떻게 더 효율적으로 운영하나"가 중요하다는 것을 보였습니다.
- T02에서 본 KV 캐시 문제를 서빙 인프라 차원에서 푼 대표작으로, 알고리즘(긴 문맥)과 시스템(서빙)이 같은 병목을 양쪽에서 공략하는 모습을 보여 줍니다.

## 한계와 주의점

- PagedAttention은 메모리 관리 기법입니다. 모델의 품질을 바꾸지 않으며, 향상되는 것은 속도와 처리량입니다.
- 블록 단위 관리에도 약간의 부기(bookkeeping) 오버헤드가 있고, 워크로드(요청 길이 분포, 공통 프롬프트 비율)에 따라 이득의 크기가 달라집니다.

## 한 줄 요약

> PagedAttention은 KV 캐시를 OS 페이지처럼 작은 블록으로 관리해 메모리 낭비를 거의 없애고 처리량을 2~4배 끌어올린다. 그렇게 만든 vLLM이 대표적 오픈 서빙 엔진이 되었다.
