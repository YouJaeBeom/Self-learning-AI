# 얼마나, 어떤 비율로 키울까 — Chinchilla

> **Training Compute-Optimal Large Language Models** — Hoffmann, Borgeaud et al. (DeepMind), 2022, NeurIPS (arXiv:2203.15556)
> 원문: https://arxiv.org/abs/2203.15556 · 분류: 바이블

Scaling Laws[2]는 "키우면 좋아진다"를 알려 줬습니다. 하지만 실무에는 한 가지 질문이 남았습니다. 한정된 연산 예산을 모델 크기와 데이터 양에 어떻게 나눠 쓸 것인가? 당시 추세는 모델 크기에 치우쳐 있었습니다. 거대한 모델을 만들고 상대적으로 적은 데이터로 학습하는 식이었죠. Chinchilla는 "둘을 동등한 비율로 함께 키워야 한다"는 교정으로, 당시 거대 모델들이 데이터를 너무 적게 먹었음(undertrained)을 드러냈습니다. 작은 Chinchilla가 더 큰 Gopher를 이긴 것이 그 증거입니다.

## 배경: 모델을 키울까, 데이터를 늘릴까

같은 연산 예산이 주어졌을 때 선택지는 둘입니다. 모델을 더 크게 만들 것인가, 아니면 같은 크기 모델에 데이터를 더 많이 먹일 것인가. 연산은 대략 모델 크기와 데이터 양의 곱에 비례하므로(`C ≈ 6·N·D`), 한쪽을 늘리면 다른 쪽을 줄여야 합니다. 당시 GPT-3, Gopher 같은 모델들은 모델 크기를 키우는 데 예산을 집중하고 데이터는 상대적으로 적게 줬습니다. 1,750억 파라미터 모델에 3,000억 토큰 정도를 준 식인데, 이것이 과연 최적이었을까요?

> 핵심 질문: 정해진 연산에서 손실을 최소화하려면, 모델 크기와 학습 토큰을 어떤 비율로 배분해야 할까?

## 핵심 아이디어: 모델과 데이터를 함께 키운다

Chinchilla의 발견은 모델 크기 `N`과 데이터(토큰) `D`를 대략 동등한 비율로 함께 키워야 최적이라는 것입니다. 연산을 두 배로 늘릴 수 있다면, 모델 크기와 데이터를 각각 약 `√2`배씩 키우는 것이 좋습니다. 모델만 키우고 데이터를 덜 주면, 그 모델의 잠재력을 다 쓰지 못한 채 학습이 끝납니다(undertrained).

> 비유 — 큰 두뇌(모델)를 만들어 놓고 공부(데이터)를 덜 시키면 잠재력을 못 씁니다. 적당한 두뇌에 충분한 공부를 시키는 편이, 큰 두뇌에 부족한 공부를 시키는 것보다 낫습니다. 머리만 좋고 안 배운 사람보다, 평범한 머리에 많이 배운 사람이 더 많은 일을 해내는 것과 같습니다.

## 수식으로 들어가기

논문은 여러 모델을 다양한 크기와 데이터로 학습해 손실을 측정하고, 다음과 같은 형태의 손실 함수를 적합했습니다.

```
L(N, D) ≈ E + A/N^α + B/D^β
```

`E`는 줄일 수 없는 자연 엔트로피, 둘째 항은 모델 크기의 한계, 셋째 항은 데이터의 한계입니다. 연산 제약 `C ≈ 6·N·D` 아래에서 이 손실을 최소화하면, 최적의 `N`과 `D`가 연산에 대해 비슷한 비율로 함께 커져야 한다는 결론이 나옵니다. 실용적으로는 파라미터 하나당 약 20개 토큰이 적절하다는 경험칙으로 요약됩니다.

```
연산 예산 C에서:  N과 D를 동등 비율로 함께 키운 지점이 최적
경험칙:  학습 토큰 D ≈ 20 × 파라미터 수 N
검증:  더 작은 Chinchilla(70B, 충분히 학습) > 더 큰 Gopher(280B, 덜 학습)
```

Chinchilla는 Gopher보다 4분의 1 작은 700억 파라미터지만, 약 1.4조 토큰으로 충분히 학습해 여러 벤치마크에서 Gopher를 능가했습니다. 크기가 아니라 충분히 먹였는가가 중요하다는 것을 실증한 것입니다.

| 모델 | 파라미터 | 학습 토큰 | 결과 |
|---|---|---|---|
| Gopher | 280B | 약 300B | 기준 |
| Chinchilla | 70B | 약 1.4T | Gopher 능가 |

## 임팩트: 왜 바이블이 되었나

- **'Chinchilla-optimal'이라는 설계 표준**을 정착시켰습니다. 이후 LLaMA 등 많은 모델이 더 많은 데이터로 충분히 학습하는 방향으로 설계됩니다.
- "모델만 키우면 된다"는 통념을 깨고 데이터의 중요성을 전면에 세웠습니다. 다음 글 FineWeb[3](데이터 큐레이션)의 동기와 직결됩니다. 데이터를 충분히 먹여야 한다면, 좋은 데이터를 충분히 확보하는 것이 다음 과제가 됩니다.
- 한정된 연산의 합리적 배분이라는, 모든 LLM 개발의 실무 기준이 되었습니다.

## 한계와 주의점

- Chinchilla 최적은 학습 비용 기준입니다. 추론을 많이 하는 서비스라면, 일부러 더 작은 모델을 더 많이 학습해(추론 비용 절감) 다르게 배분하기도 합니다. LLaMA가 파라미터당 토큰을 훨씬 늘린 이유입니다. 한 번 학습해 수십억 번 추론한다면, 작고 빠른 모델이 유리합니다.
- 결론은 충분한 고품질 데이터가 있다는 전제에 기댑니다. 데이터가 부족하거나 질이 낮으면 이야기가 달라집니다(FineWeb[3]).
- 멱법칙처럼 관측 범위의 경험칙이라, 극단적 규모에서는 재검증이 필요합니다.

> Chinchilla는 모델과 데이터를 동등 비율로 키워야 최적임을 보여(작은 Chinchilla가 큰 Gopher를 능가), 기존 거대 모델이 데이터를 덜 먹었음을 드러내고 설계 표준을 바꿨다.
